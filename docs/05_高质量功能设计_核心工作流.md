# GT Office 高质量功能设计（核心工作流）

## 1. 设计目标

本文件仅保留“可直接驱动实现与验收”的信息：
1. User Story（前置条件、主流程、异常流程）。
2. FR/NFR（可验收）。
3. 命令/事件/错误码引用点（与 `docs/06` 对齐）。
4. 验收用例（正常流 + 异常流）。

## 2. 设计门槛（进入编码前）

每个功能点必须满足：
1. 有 `FR` 与 `NFR`，且能映射到测试。
2. 至少 1 条 User Story，含前置条件/主流程/异常流程。
3. 契约字段明确（命令 req/resp、事件 payload、错误码）。
4. 数据影响明确（配置字段、表结构、状态机）。
5. 至少 1 条正常流 + 1 条异常流验收用例。

## 4. User Stories（活跃）

### US-26 多 Agent MCP 桥接通信（T-171）

- FR：
  1. manager 可派发任务到一个或多个执行 agent。
  2. 执行 agent 可回传 `status/handover` 给 manager。
  3. 支持 Codex/Claude/Gemini/Qwen 的 MCP 配置手动安装，且安装入口必须与 CLI 安装分离。
  4. Agent 供应商页面进入时必须先检测本机 CLI 是否已安装；已安装项隐藏安装入口，仅对未安装项展示安装 CTA。
  5. agent 间通信默认必须走 GT Office MCP，不依赖 manager 再额外提示“请用 MCP 回复”。
- NFR：
  1. 桥接仅监听 `127.0.0.1` 且 token 鉴权。
  2. 请求超时与重试策略稳定，错误码可机读。
  3. 安装流程幂等，重复执行不产生重复配置。
  4. 支持“GT Office 桌面端运行于 Windows、agent/MCP 运行于 WSL”的混合部署；发送链路必须优先选取与目标 workspace 同源的 runtime/directory 状态根。
  5. 正常协作回复必须收敛为最小结果，不在会话中回显大段 tool guidance、环境说明或完整诊断 JSON。
- 前置条件：GT Office 已启动；目标 `workspace_id` 有可用 runtime。
- 主流程：
  1. 用户打开 Agent 供应商页面时，前端并行查询各 CLI 的安装状态与前置环境状态。
  2. 每个 agent 卡片同时展示 CLI 状态和“增强服务”入口；卡片层不再直接暴露 MCP 术语。
  3. 用户点击“增强服务”后，系统打开独立弹窗，按 `MCP / Skills` tab 分类展示扩展能力。
  4. `MCP` tab 中当前以 `GT Office Agent Bridge` 服务项承载安装入口；`Skills` tab 作为后续扩展占位。
  5. 用户点击“安装 MCP”后，桌面端仅为当前 agent 定向写入 MCP server 配置与默认协作 policy 文件，不得同时修改其他 agent 的配置。
  6. 协作前可调用 `gto_get_agent_directory`，拿到当前 workspace 的 agent 与 `self` 身份；若未显式传 `workspace_id`，系统可从 env 或最近 snapshot 自动推断。
  7. manager 按目录结果中的 `agent_id` 调用 `gto_dispatch_task`；该动作会将文本写入目标 agent terminal，并按 runtime 的 `submitSequence` 自动提交执行。
  8. 系统派发并返回 `batchId + per-target status`。
  9. 执行 agent 默认把“与其他 GT Office agent 通信”解释为使用 `gto_report_status/gto_handover` 回传；这两类 tool 仅写入协作消息，不负责向目标 terminal 注入命令。
  10. `sender_agent_id` 默认按“显式参数 -> terminal env -> 当前 cwd/目录快照自识别”解析；仅在三者都失败时才报错。
  11. sender agent 可通过 `gto_list_messages` 读取自己的 inbox，拿到执行 agent 通过 `status/handover` 回传的正文内容。
- 异常流程：
  1. 桥接不可达 -> `MCP_BRIDGE_UNAVAILABLE`。
  2. 鉴权失败 -> `MCP_BRIDGE_AUTH_FAILED`。
  3. 目标离线 -> `AGENT_OFFLINE`。
  4. bridge 不可达但本地 `directory.json` 存在 -> 目录查询仍返回最近快照，health 返回降级诊断。
  5. 当前 terminal 缺失 `GTO_AGENT_ID` -> 继续尝试按 cwd 匹配目录中的 agent；仍失败才返回 `MCP_INVALID_PARAMS`。
  6. 执行 agent 只在自己终端直接打印回复、未走 `gto_report_status/gto_handover` -> sender inbox 中不会出现该内容。
- 验收标准：
  1. 至少一个 CLI 成功完成“目录查询 -> 派发 -> handover 回传”闭环。
  2. 一发多目标返回独立结果且与任务中心一致。
  3. 未显式传 `sender_agent_id` 时，`handover/status` 仍可通过当前 agent terminal env 或 cwd/目录自识别成功发送。
  4. manager 不再需要额外提示“请用 MCP 回复”，执行 agent 也能默认走 bridge 回传。
  5. sender 在派发后可通过 `gto_list_messages` 拉到回复内容，而不必依赖桌面 UI 事件流。
  6. 页面进入后已安装 CLI 不再显示安装按钮，未安装项安装按钮在亮暗主题下都具备清晰可见的 CTA 对比度。
  7. 页面中的 MCP 状态必须收口到“增强服务”弹窗；弹窗内需明确区分“未安装 / 已预配置待 CLI / 已安装”，且 bridge 启动或 CLI 安装过程不得再偷偷自动写入 MCP 配置。
  8. CLI 安装成功必须以后置真实校验为准：至少要能解析到受支持的可执行文件，并通过一次实际版本探测；不得仅以安装脚本返回码为“已安装”依据。

### US-27 外部通道接入与多 Agent 路由闭环（T-172）

- FR：
  1. 支持 `feishu/telegram` connector 账号配置、健康检查与 route 绑定。
  2. 支持 `feishu/telegram` webhook 入站；其中 Feishu 需额外支持桌面端长连接模式配置与探测。
  2. 入站消息必须走统一路由与准入策略（`pairing/allowlist/open/disabled`）。
  3. 必须有幂等去重与统一外部链路事件。
  4. 目标 `role:<role_key|role_id>` 可解析为可投递 agent 集合。
  5. 回复链路必须采用“双源观察”：
     - `RenderedScreenSnapshot` 负责 preview、ready/finalize 边界与交互 prompt。
     - provider `session log` 负责高置信最终正文，当前 v1 支持 Claude/Codex。
     - session log 不可用时，自动回退到 rendered-screen，再回退到 VT 文本。
- NFR：
  1. 外部链路失败不影响主任务中心与终端主路径。
  2. 幂等命中 O(1) 查询。
  3. 渲染快照上报与 terminal 显示解耦，失败不阻塞执行。
  4. Windows 生产包下，connector polling / health probe / outbound send 等后台子进程不得弹出系统控制台窗口。
  5. 非 Windows 平台保持现有子进程行为，不因该修复引入新的启动副作用。
- 前置条件：已配置 channel 绑定；目标 workspace/runtime 在线。
- 主流程：
  1. 用户在 GT Office 内选择 channel、配置 connector 账号，并在平台侧完成 Bot / 权限 / 事件订阅。
  2. 系统执行 connector health probe，并返回 bot identity、callback 匹配状态与 runtime 状态。
  3. 用户保存 route binding 与 access policy。
  4. webhook 或长连接入站归一化为 `ExternalInboundMessage`。
  5. 路由匹配 + 准入校验 + 幂等判重。
  6. 写入已绑定 live terminal session 并执行。
  7. 前端持续上报 rendered-screen；后端基于 rendered-screen 生成 preview、interaction prompt 与 finalize 边界。
  8. 后端按 runtime 的 `tool_kind + resolved_cwd + providerSession + injected_input` 绑定 provider session log；Claude/Codex 轮询 session log 作为 finalize 正文主源。
  9. Codex 绑定优先级必须为：`providerSession.logPath > providerSession.providerSessionId > cwd + bind 时间窗 + prompt anchor > 同 cwd 最新活跃日志`。
  10. Codex assistant transcript 允许包含 commentary；若 preview 已通过 rendered-screen 发出前缀内容，则 finalize 只补发 transcript 的新增尾段，不重复整段正文。
  11. Telegram 交互 prompt 需要支持两类回流：
     - 语义按钮：`gto:<submit_text>`
     - 方向键按钮：`gto-key:up|down|enter|esc|tab`
- 异常流程：
  1. route miss -> `CHANNEL_ROUTE_NOT_FOUND`。
  2. pairing 未通过 -> `CHANNEL_PAIRING_REQUIRED`。
  3. connector health 未通过 -> 不允许完成最终绑定。
  4. session log 缺失、歧义或 provider 未支持 -> 自动回退到 rendered-screen / VT fallback，不额外拉起 headless CLI。
  5. rendered-screen 置信度不足且无 session log 正文 -> 正文静默，不发送噪声回复。
  6. session log 文件实时写入、半行 JSON、truncate/rotate -> 解析器必须增量容错，且读取过程不得阻塞前端交互线程。
- 验收标准：
  1. 双通道均可完成“connector 配置 -> health probe -> route 绑定”闭环。
  2. 双通道入站均可进入统一派发链路。
  3. 同一幂等键重放 3 次只派发 1 次。
  4. 权限确认/菜单内容不混入正文；Telegram 语义按钮与 `gto-key:*` 方向键回调都必须回写同一 live session。
  5. Claude/Codex 在 session log 可用时，final body 优先来自 session log；preview 仍来自 rendered-screen。
  6. Codex 在同 cwd 存在多个 rollout 时，显式 `providerSessionId/logPath` 能稳定命中当前对话，不串到其他会话。
  7. Codex finalize 若 preview 已覆盖前缀，只发送新增尾段；delta 为空时不重复外发正文。
  8. Feishu v1 仅展示交互提示文本，不提供远程方向控件。
  9. Windows 安装包在存在 Telegram polling 账号时启动后不再反复弹出 `cmd`/控制台窗口。
  8. Linux/macOS 开发与生产环境下 connector 健康检查、发送与轮询行为保持不变。

### US-30 Agent 供应商配置中心（T-171）

- FR：
  1. `ai` 设置页必须升级为统一的 Agent 配置中心，而不是仅展示安装卡片。
  2. Claude Code 必须支持供应商切换、模型覆盖、API Key 托管，并强制走“预览 -> 校验 -> 确认 -> 应用 -> 审计”。
  3. Codex CLI 与 Gemini CLI 在 v1 只提供安装状态、官方文档、本地配置路径和轻量说明，不暴露复杂供应商表单。
  4. Claude 供应商必须内置精选预设与新手引导，覆盖充值入口、API Key 获取入口和推荐模型。
  5. Claude API Key 只能保存到系统凭据库；`.gtoffice/config.json` 只允许保存 `secretRef` 与非敏感配置。
  6. Claude/Codex/Gemini 配置向导中的“无输入纯过渡步骤”必须自动跳过或默认落在首个可操作步骤，禁止用户为继续流程额外点击一次“下一步”。
  7. Claude 每次成功应用后的供应商配置都必须保存到数据库，并在 Agent 供应商列表中的独立“已保存供应商”子入口里可查看、可直接切换；“配置向导”和“已保存供应商”禁止做成同一弹窗内 tab。
  8. Claude 选择 `official` 时，必须把当前生效配置切换为官方档案，并用该官方档案覆盖此前生效的第三方供应商状态；预览、快照与已保存记录中必须回填 Anthropic 官方基线 `provider/baseUrl/model/authScheme`，不能继续显示上一个第三方供应商残留值。
- NFR：
  1. 设置页在 `375/768/1024/1440` 下无横向滚动。
  2. Claude 预设切换与预览反馈必须在 1 个面板内完成，避免用户在多个弹窗之间来回跳转。
  3. 相同逻辑必须抽成 shared 组件；Claude / Codex / Gemini 目录必须分开。
  4. Claude 配置应用后必须同步受管环境变量到系统级 `~/.claude/settings.json`，并保留已有 `mcpServers`、权限及非受管 env 字段；不要求热切换已运行会话。
- 前置条件：已打开 workspace；若配置 Claude 供应商，则本机可安装或已安装 Claude CLI。
- 主流程：
  1. 用户进入 Agent 配置中心，前端读取 `ai_config.read_snapshot`，同时拿到 3 个 Agent 的安装状态、Claude 当前配置、预设目录和轻量引导信息。
  2. 顶部统一展示 3 个 Agent 状态卡，用户可快速切换到 Claude/Codex/Gemini。
  3. 进入 Claude 后，按 `安装检查 -> 选择供应商 -> 充值/API Key 引导 -> 模型/高级项 -> 预览并应用` 向导完成配置。
  4. 用户点击预览后，后端返回 `previewId + normalizedDraft + maskedDiff + warnings + secretRefs`。
  5. 用户确认应用后，后端写入工作区配置、系统凭据库、`~/.claude/settings.json`、Claude saved provider 数据表和 `ai_config_audit_logs`，并发出 `ai_config/changed`。
  6. 用户从 Claude 卡片下的“已保存供应商”子入口进入列表页时，可以查看所有历史应用成功的 Claude 配置，并直接切换当前生效项；切换后工作区配置、live settings 与 active saved provider 必须保持一致。
  7. 用户在 Claude 已保存供应商列表中点击“编辑配置”时，界面必须复用 Claude 详情参数配置页；预览/应用链路需继续走 `preview -> apply -> audit`，并优先复用目标 saved provider 原有的 `savedProviderId` 与托管密钥，不得额外新建重复记录。
  8. 用户启动新的 Claude 会话时，无论是 GT Office 内新终端还是系统级独立 Claude CLI，都会读取同一份 live settings；已运行会话不热切换。
- 异常流程：
  1. workspace 未打开 -> 不允许进入配置流，提示先打开工作区。
  2. Claude 预设缺少 API Key 且无已有 `secretRef` -> `AI_CONFIG_INVALID`。
  3. `previewId` 过期或不存在 -> `AI_CONFIG_PREVIEW_NOT_FOUND`。
  4. 系统凭据库写入失败 -> `AI_CONFIG_SECRET_FAILED`。
  5. `~/.claude/settings.json` 读写失败或结构非法 -> `AI_CONFIG_LIVE_SYNC_FAILED`。
  6. 切换到不存在的 saved provider -> `AI_CONFIG_SAVED_PROVIDER_NOT_FOUND`。
- 验收标准：
  1. Claude 至少 1 个精选预设能完成“预览 -> 应用 -> 更新 `~/.claude/settings.json` -> 启动新会话生效”闭环。
  2. `.gtoffice/config.json` 中不出现 API Key 明文，只出现 `secretRef`。
  3. Codex/Gemini 页面不出现空白复杂表单，且能提供可点击的官方文档与本地配置路径说明。
  4. Claude 向导支持返回上一步、跳过引导、inline 错误反馈和成功提示。
  5. 已安装 CLI 打开向导时默认落在首个可编辑步骤；在 Claude 供应商模式页选择 `official/custom` 后应立即进入对应下一步，不再额外停留在无输入页面。
  6. Claude 卡片必须同时提供“配置向导”和“已保存供应商”两个子入口；saved provider 列表页至少展示 2 条已保存配置，并支持一键切换当前生效供应商；切换时只更新激活标记，不得因为激活状态变化重排列表顺序。
  7. Claude saved provider 列表必须支持“编辑配置”；从列表进入编辑时应复用详情参数配置页，并在应用后继续使用原 `savedProviderId` 更新该条记录，而不是新增一条重复配置。

### US-28 样式系统工程化重构（T-175）

- FR：
  1. 样式入口统一为 `src/styles/index.scss`。
  2. 样式分层：`tokens/foundations/features/utilities`。
  3. 历史变量收敛为设计令牌并保留兼容别名。
- NFR：
  1. 分阶段可发布，迁移过程持续可构建。
  2. 不引入不可控全局副作用。
- 前置条件：`docs/07` 已允许 `sass` 依赖。
- 主流程：
  1. 搭建 SCSS 骨架并切换入口。
  2. 按 feature 迁移并去重。
  3. 执行最小验证链路。
- 异常流程：
  1. 构建失败 -> 回滚到最近可编译状态并定位层级冲突。
  2. 视觉偏差超阈值 -> 补兼容 token，延后视觉重构。
- 验收标准：
  1. 不再新增业务 `.css` 入口。
  2. `npm run typecheck`、`npm run build:web`、`cargo check --workspace` 可通过。

### US-34 响应式样式单位重构（T-187）

- FR：
  1. `apps/desktop-web/src` 业务 UI 的固定 `px` 单位必须逐步迁移为 `rem / clamp / % / fr / dvh` 等响应式单位。
  2. 所有尺寸必须先进入统一的 semantic token，不允许 feature 继续直接写死尺寸语义。
  3. Shell 层负责 viewport 级响应式，feature 内部优先使用 container query 完成窄宽度重排。
  4. Git feature 作为 Phase 4 首个样板，必须先打通 `operations / history / diff / graph` 四类界面的响应式迁移链路。
  5. `GitPane.tsx` 与 `GitGraphView.tsx` 中的内联尺寸输出不得再拼接 `px` 字符串。
- NFR：
  1. 默认窗口和默认字体档位下，改造前后视觉比例不得出现明显跳变。
  2. Windows `100% / 125% / 150%` 与 macOS 标准屏 / Retina 下均需保持控件可读、可点、无裁切。
  3. 迁移过程必须持续可构建，且允许分 feature 渐进推进。
  4. 高密度列表和虚拟列表在切换为 rem 输出后不得出现高度错位或滚动抖动。
- 前置条件：
  1. `docs/09_响应式样式单位重构方案.md` 已明确 token、断点、单位转换和专项验收矩阵。
  2. Shell 层 Phase 3 重构已完成，可作为 feature 迁移基线。
- 主流程：
  1. 先清理目标 feature 的 `px` 字面量和 `px-to-rem(...)` 过渡写法，统一切到语义 token 与 `rem()` helper。
  2. 将 TSX 内联高度、位移、虚拟列表尺寸输出改为基于根字号的 rem 表达。
  3. 为 feature 根容器补充 container query，窄宽度下优先换行、堆叠、压缩非关键列。
  4. 逐个完成 `operations / history / diff / graph` 的构建验证与定向审计，再推广到下一个热点模块。
- 异常流程：
  1. 单位替换后出现视觉跳变或高度链失稳 -> 先回到等比 rem 映射，再延后 `clamp()` 与流式增强。
  2. 虚拟列表因行高与位移语义不一致出现错位 -> 统一由同一根字号换算路径输出高度和偏移。
  3. container query 与旧 viewport 断点冲突 -> 保留 shell 断点职责，feature 内删除重复规则。
- 验收标准：
  1. `cd apps/desktop-web && npm run build` 通过。
  2. `rg "\b\d+(\.\d+)?px\b" apps/desktop-web/src/features/git apps/desktop-web/src/styles/features/git/_pane.scss` 返回 `0`。

  3. 同一套文件视觉语义需在三个位置一致：文件树、文件搜索结果、编辑器标签。

- NFR：
  1. 保持虚拟列表性能与键盘交互语义不回退。
  2. 亮/暗主题均需保证可读性与边界对比（focus/selected 对比清晰）。
  3. 窄宽度下不出现横向滚动；格式 badge 可自适应隐藏。

- 主流程：
  1. 用户进入“文件管理”，文件树行态呈现统一图标胶囊与更清晰层级。
  2. 用户搜索文件或内容时，结果列表使用与文件树一致的格式图标语义。

  3. 文件树/搜索结果/编辑器标签三处的图标语义一致，且未出现交互回归。

### US-32 全局任务派发浮层（T-175）

- FR：
  1. 任务中心中“选择 agent + 发送任务”两项核心能力必须抽成可全局唤起的独立组件。
  2. 用户在任意主界面下都可通过默认快捷键打开该浮层，无需先切到任务中心导航。
  3. 默认快捷键必须可在设置中心修改，并在设置页内提供恢复默认能力。
  4. 浮层必须支持透明度调节，兼顾任务编写可读性与当前界面上下文保留。
  5. 浮层沿用当前任务草稿、agent 选择与 `@` 文件引用能力，不能新增一套割裂的数据流。
- NFR：
  1. 快捷键录制过程不能误触发现有全局快捷键。
  2. 浮层需支持键盘关闭与发送；至少支持 `Esc` 关闭、`Mod+Enter` 发送。
  3. 样式需保持“专业桌面工作台”气质，避免装饰性过强的浮夸视觉。
- 前置条件：应用已启动；若要真正派发任务，当前存在活动 workspace。
- 主流程：
  1. 用户在任意界面按下全局任务派发快捷键。
  2. 系统以 portal 打开任务派发浮层，自动聚焦任务输入区，并展示当前快捷键与透明度控制。
  3. 用户选择一个或多个 agent，输入 markdown 任务并发送。
  4. 浮层复用任务中心既有派发链路完成发送，并延续草稿持久化与状态提示。
  5. 用户在设置中心的“快捷键与浮层”页调整快捷键或透明度后，系统立即生效并持久化到 settings。
- 异常流程：
  1. 无活动 workspace 时尝试发送 -> 保持浮层可见，并复用任务中心既有错误提示。
  2. 快捷键录制时仅按修饰键 -> 不保存，继续等待完整组合键。
  3. 设置持久化失败 -> 保留当前会话内交互，不阻断用户继续派发。
- 验收标准：
  1. 在 `stations/files/git/channels` 等主界面中均可通过同一快捷键打开浮层。
  2. 默认快捷键为 `Mod+Shift+K`，且在设置中心修改后立即生效。
  3. 调整透明度后再次唤起浮层，视觉透明度保持最新设置值。
 4. 浮层内可完成多 agent 选择、markdown 任务输入、`@` 文件引用与发送闭环。

### US-33 通信通道侧栏用户化简化（T-175）

- FR：
  1. “通信通道”侧栏必须改为面向普通用户的消息视图，不再暴露运行状态、Webhook 地址、路由绑定、准入白名单、待配对、幂等缓存等运维指标。
  2. 出入站消息必须先按“两端通信关联键（外部端点 + 内部 target agent）”分组，并在顶部 Tab 中切换会话，禁止把多个通道/会话混排在同一消息流里。
  3. 消息气泡仅展示正文（失败时可附错误详情）；删除“最近外部事件”标题与单条 `时间/channel/status` 元信息展示。
  4. 样式需采用 Apple 风格的轻玻璃工作台语言，但避免卡片套卡片；深色模式下气泡背景不得过亮。
  5. 顶部会话导航必须是非气泡的扁平 tab，标签文案使用“channel bot 名称 - agent 名称”，禁止展示长 peer/account id。
- NFR：
  1. 消息列表在桌面宽度下保持高信息密度，移动窄宽度下自动换行，不出现横向滚动。
  2. 前端不得通过解析展示文案来推断 channel；事件载荷若已有通道上下文，应直接透传。
- 前置条件：存在外部通道事件流；至少支持 Telegram/Feishu 的入站或出站消息展示。
- 主流程：
  1. 用户打开“通信通道”侧栏。
  2. 系统展示“顶部会话 Tab + 当前会话消息流”主面板，而不是多组运行状态卡片。
  3. 每条消息仅展示正文摘要（失败可附错误详情），会话级别信息在 Tab 上体现。
- 异常流程：
  1. 当前无外部事件 -> 展示简洁空状态，不回退显示运维面板。
  2. 某些历史出站事件缺失 channel -> UI 使用保守回退值，但后续新事件必须透传真实 channel。
- 验收标准：
  1. 侧栏中不再出现“Webhook 运行状态、地址和最近事件”“运行中”“刷新”“路由绑定”“准入白名单”“待配对”“幂等缓存”等文案。
  2. 同一“外部端点 + target agent”的消息稳定进入同一 Tab；不同通道或不同 target agent 不再混排。
  3. 消息气泡不再显示时间/channel/status 元信息，仅展示正文与失败详情。
  4. `npm run build` 通过。

### US-29 角色管理侧栏真实数据与编辑收敛（T-175）

- FR：
  1. 角色管理侧栏列表项仅保留编辑入口，不再直接暴露删除按钮。
  2. 新增角色与编辑角色必须复用同一弹窗。
  3. 删除仅允许在编辑态触发。
  4. 角色管理列表必须基于 SQLite `agents + agent_roles` 真实数据，而非前端写死 seed。
- NFR：
  1. 旧数据库升级不要求手工迁移；缺失字段需自动补 schema。
  2. 角色编辑后无需重启前端即可刷新侧栏与中央画布。
- 前置条件：已绑定 workspace；Tauri runtime 可访问本地 SQLite。
- 主流程：
  1. 前端进入角色管理页时，按当前 `workspaceId` 拉取 `agent.role_list` 与 `agent.list`。
  2. 侧栏列表展示 agent 实例，点击编辑按钮打开复用弹窗。
  3. 新增/编辑提交后分别调用 `agent.create` / `agent.update`，成功后重新拉取列表。
  4. 编辑态点击删除后调用 `agent.delete`，成功后刷新列表并关闭弹窗。
- 异常流程：
  1. 数据库 schema 旧版本缺列 -> `ensure_schema()` 自动补列。
  2. role 不存在 -> 后端返回 `invalid argument: role_id not found`，前端保持现状不落脏数据。
- 验收标准：
  1. 列表项不再显示删除按钮，编辑入口可正常打开弹窗。
  2. 新增、编辑、删除各执行一次后，刷新页面仍能从数据库读到一致结果。
  3. `npm run build` 与 `cargo check --workspace` 通过。

  2. 点击 macOS 前端交互控件 -> 仅触发控件动作，不进入前端自绘拖拽态。
- 验收标准：

  2. macOS 下主内容区按钮单击无需双击。
  3. Windows 下标题栏拖拽与双击最大化行为保持不变。
