# GT Office 高质量功能设计（核心工作流）

## 1. 设计目标

本文件将需求从“能力描述”升级为“可执行设计”，用于指导研发、测试、验收三方协同。

目标：

1. 明确每个核心功能的用户价值、输入输出、状态变化。
2. 固化主流程与异常流程，避免实现偏差。
3. 为接口设计、测试用例、迭代拆分提供直接依据。

## 2. 信息架构（IA）

`T-010` 已确认采用 Agent First（角色优先）布局，不再采用传统编辑器中心模型。

### 2.2 区域职责（Agent First）

1. 顶部控制带：全局级控制与调度，不承载局部业务细节。
2. 主导航轨道：一级能力入口（角色、任务、文件、Git、Hook、通道、策略、设置）。
3. 左侧业务面板：当前一级能力的主操作面板（可切换内容，不切换布局骨架）。
4. 中央角色画布：核心执行面，显示 Agent 角色宫格窗口，每格内嵌终端命令行与会话控制。
   - 输入交互采用终端窗口原位键盘输入，回车直接执行命令。
5. 状态栏：展示系统态、工作区态、资源态、策略态。

### 2.3 扩展性约束（面向百级 Agent）

1. 角色矩阵必须虚拟化渲染，只渲染可视区。
2. 终端输出采用分级订阅：前台实时、后台降频或摘要；终端渲染使用 xterm 并通过增量流式写入，避免高频整树重渲染。
3. 角色窗口与 PTY 会话生命周期解耦，避免 UI 卡顿影响执行。
4. 所有区域数据通过统一事件总线更新，避免点对点耦合。
5. 多工作区切换时，所有面板必须按 `workspace_id` 原子切换上下文。

### US-26 多 Agent MCP 桥接通信（T-171）

- FR：
  1. 提供独立 MCP 工具（与主工程解耦），支持 manager agent 通过统一工具把任务派发给一个或多个执行 agent。
  2. MCP 工具必须复用任务中心派发语义（`task_dispatch_batch`）与通道语义（`channel_publish`），禁止旁路写入。
  3. 执行 agent 完成任务后可通过 MCP 工具向 manager 发送 `status/handover` 汇报，形成闭环链路。
  4. 安装 GT Office 时自动写入主流 CLI Agent 的 MCP 配置：Claude Code、Codex、Gemini CLI、Qwen CLI。
  5. 工具侧支持 `health` 检查与结构化错误输出，便于运维排障。
- NFR：
  1. MCP 工具与 GT Office 主工程分目录维护（`tools/`），通过本地桥接协议解耦，避免跨模块硬耦合。
  2. 桥接服务仅监听 `127.0.0.1`，并启用 token 校验，拒绝未授权调用。
  3. 同一请求具备超时与可重试策略，失败返回稳定错误码（`MCP_BRIDGE_*`）。
  4. 自动安装流程要求幂等，多次执行不应重复污染配置文件。
- 前置条件：GT Office 已启动并已激活目标 `workspace_id`；目标 agent runtime 已在线。
- 主流程：
  1. manager agent 通过 CLI MCP 客户端调用 `gto_dispatch_task`，输入 `workspace_id/targets/title/markdown`。
  2. MCP 工具经本地桥接调用任务中心派发能力，返回 `batch_id + per-target task_id/status`。
  3. 执行 agent 在完成后调用 `gto_report_status` 或 `gto_handover`，汇报进度、阻塞与下一步建议。
  4. manager 端通过通道事件与任务状态视图回看完整链路。
- 异常流程：
  1. 桥接不可达 -> 返回 `MCP_BRIDGE_UNAVAILABLE`，并提示检查 GT Office 运行状态。
  2. token 校验失败 -> 返回 `MCP_BRIDGE_AUTH_FAILED` 并拒绝处理。
  3. 目标 agent 离线 -> 返回 `AGENT_OFFLINE`，保留失败目标明细。
  4. CLI 未安装或配置文件不可写 -> 自动安装记录失败详情，但不阻断主程序启动。
- 验收标准：
  1. Codex/Claude/Gemini/Qwen 任一 CLI 成功加载 MCP 工具并可调用 `gto_dispatch_task`。
  2. 同一次调用派发到 N 个目标时，返回 N 条独立结果，状态与任务中心一致。
  3. 执行 agent 的 `status/handover` 能回传到 manager 对应通道。
  4. 自动安装重复执行 3 次后配置仍保持单一有效条目（无重复 server 节点）。

### US-27 外部通道接入与多 Agent 路由闭环（T-172）

- FR：
  1. GT Office 提供统一外部通道接入层，首期支持 `feishu` 与 `telegram` 两类 webhook 入站。
  2. 桌面端必须内置 webhook runtime（仅监听 `127.0.0.1`），并输出可配置的 Feishu/Telegram 回调地址。
  3. 外部入站必须通过统一路由绑定（`channel + account + peer -> workspace_id + role_selector`）进入任务中心，禁止绕过任务派发主链路。
  4. 会话隔离默认使用 `per-channel-peer`，同一外部来源需稳定映射同一会话键，防止多人上下文串扰。
  5. 准入策略默认 `pairing`，未授权来源不得触发任务派发；授权通过后可进入正常派发流程。
  6. 入站消息必须支持幂等去重，重复投递不得重复创建任务。
  7. 全链路需发射外部事件：`external/channel_inbound`、`external/channel_routed`、`external/channel_dispatch_progress`、`external/channel_reply`、`external/channel_error`。
  8. 提供 `system.gto_doctor` 诊断入口，输出路由、准入、幂等状态快照与修复建议。
  9. 设置页提供 Telegram/Feishu 配置引导与可执行操作入口（路由绑定、准入策略、identity 授权、回调地址复制）。
  10. 路由目标支持岗位选择：`targetAgentId` 可写入 `role:<role_key|role_id>`，入站时自动解析岗位目标集合（岗位 agent 档案 + `agent.runtime_register.roleKey` 在线实例）并派发。
  11. 通道连接器需提供账户管理与健康检查命令：`channel_connector_account_upsert`、`channel_connector_account_list`、`channel_connector_health`、`channel_connector_webhook_sync`（首期 Telegram 落地，Feishu 预留）。
  12. Telegram connector 需支持 `getMe/getWebhookInfo/setWebhook` 健康闭环，允许将 runtime 生成的 webhook URL 一键同步到 Telegram Bot。
  13. 外部派发结果除 `external/channel_reply` 外，还需发射 `external/channel_outbound_result` 供任务中心做统一回执展示。
  14. 外部入站触发 Agent 执行后，系统需自动将 Agent 响应回传至来源 channel；若终端为流式输出，需采用“节流预览（同一消息编辑）+ 结束态最终收敛”策略，禁止逐 chunk 新消息刷屏。
  15. Channel 管理界面必须采用 `channel -> bot(accountId) -> route(target)` 分层展示；同一 channel 下多个 bot 必须独立可见并可编辑。
  16. Station 卡片需展示“被哪些 channel/bot 绑定”的摘要标签，且仅展示命中当前 station 的绑定。
  17. 设置页 Channel 绑定列表需展示 `bot 名称 - 绑定 agent - 创建时间` 摘要（`bot 名称` 取自健康检查结果并持久化），并提供行级健康检查动作（图标按钮）。
  18. 外部回复正文提取必须基于前端 xterm 已渲染可视屏幕的 `RenderedScreenSnapshot`，不得再以 PTY 原始 chunk/buffer 作为正文提取事实源。
  19. 前端需为每个 terminal session 维护单调递增的 `screenRevision`，并在终端输出写入后按节流窗口采集可视文本行快照并上报后端。
  20. 后端只允许对高置信 rendered-screen 候选正文做 preview/finalize；当无法区分正文与状态/工具/placeholder 时必须静默，不允许回退整屏文本。
  21. 当 rendered-screen 中出现权限确认、编号菜单、`/command` 菜单等交互态时，系统必须将其从正文中剥离，并以 Telegram inline keyboard 形式单独下发快捷操作。
  22. Telegram 按钮回调必须重新写回当前绑定的 live terminal session，并复用同一 reply session；按钮选择不得把原始问题的 reply session 顶掉。
- NFR：
  1. 业务逻辑收敛在核心服务层（`vb-task` + Tauri command 编排），不在 UI 层实现路由/准入判定。
  2. 外部通道失败不得影响既有任务中心与终端会话主流程。
  3. 幂等命中路径必须在 O(1) 查询内返回，避免平台重试风暴放大系统负载。
  4. webhook runtime 需具备基础稳定性保护：请求读取超时、按通道限流、异常计数与最近错误快照。
  5. 连接器凭据不直接落盘明文，配置层仅保存 `*_ref` 引用；系统密钥后端不可用时允许进程内回退（开发态）并在健康检查中显式暴露风险。
  6. rendered-screen 采样必须与 terminal 主显示解耦，采样失败不得影响 live terminal 执行与显示。
  7. rendered-screen 上报需按 session 节流并保证 `screenRevision` 单调，避免乱序快照污染回复判定。
  8. Telegram 交互按钮必须与当前 TUI 状态同步；菜单消失后按钮需及时失效或移除，禁止保留陈旧快捷键。
- 前置条件：目标 `workspace_id` 已存在；配置阶段仅需岗位可选，消息派发时系统按岗位解析可投递目标（agent 档案或在线 runtime）。
- 主流程：
  1. GT Office 启动时拉起 webhook runtime，并生成 `feishu` / `telegram` 回调 URL。
  2. 外部平台将消息推送到 GT Office 通道入口。
  3. 系统解析并归一化为 `ExternalInboundMessage`，生成/提取 `idempotency_key`。
  4. 系统完成路由匹配并进行准入校验。
  5. 校验通过后系统调用 `task_dispatch_batch` 派发到目标 Agent。
  6. 目标消息直接写入已绑定的 live terminal session 并提交执行。
  7. 前端 xterm 在输出写入后采集最近已渲染屏幕快照，上报后端参与正文候选提取。
  8. 后端基于连续 `RenderedScreenSnapshot` 做 diff、稳定窗口与高置信正文判定；若检测到权限确认或菜单交互，则生成结构化 interaction prompt。
  9. Telegram connector 将 interaction prompt 渲染为 inline keyboard；用户点击按钮后通过 `callback_query` 把选择写回当前 live terminal session。
  10. 正文 preview/finalize 与 interaction prompt 更新并行推进，但菜单/权限块不得混入正文消息。
  11. 任务中心与通道事件同步更新外部链路状态。
- 异常流程：
  1. 路由未命中 -> 返回 `route_not_found`，发射 `external/channel_error`。
  2. 准入未通过且策略为 pairing -> 返回 `pairing_required` 并附配对码。
  3. 准入未通过且策略为 allowlist/disabled -> 返回 `denied`。
  4. 幂等命中 -> 返回 `duplicate`，不重复派发任务。
  5. 派发失败 -> 返回 `failed` 并附错误详情。
  6. webhook token 不匹配 -> 返回 `401` 并拒绝处理。
  7. rendered-screen 采样失败或后端置信度不足 -> terminal 正常执行，但该条外部回复正文静默不发送。
  8. Telegram 按钮回调丢失或 connector 不支持 callback -> live terminal 继续运行，但该轮快捷操作降级为用户手动发送文本输入。
- 验收标准：
  1. 飞书与 Telegram 两类入站均可进入统一路由与派发链路。
  2. 同一 `idempotency_key` 重放 3 次只会产生 1 条新任务。
  3. 未授权来源不会触发任务派发，pairing 模式会返回稳定配对码。
  4. 任务中心可看到外部链路事件并与派发结果一致。
  5. 当同一 channel 存在多个 bot（不同 `accountId`）时，设置页能分组展示每个 bot 的绑定目标，且 Station 卡片可看到对应 bot 绑定摘要。
  6. 绑定列表每行可触发健康检查，并可展示 `bot-agent-createdAt` 结构化摘要；当行内无绑定目标时使用纯空态占位，不展示“未绑定目标规则”文本。
  7. 外部入站命中的 live terminal session 在执行过程中，前端能持续上报 `RenderedScreenSnapshot`，且不影响终端交互。
  8. `thinking / working / running command / tool/status / prompt / placeholder` 不会被误发到 Channel 正文；当无法明确判定正文时系统静默。
  9. 当 Codex/Claude TUI 出现编号菜单、权限确认或 ` /command ` 菜单时，Telegram 能展示对应快捷按钮，并且点击按钮后仍由当前 live terminal session 执行。
  10. 菜单标题、选项、权限确认文本不会混入 Telegram 正文回复；正文只保留真正 assistant 响应。

### US-28 样式系统工程化重构（T-175）

- FR：
  1. Web 端样式入口统一为 `src/styles/index.scss`，禁止业务组件继续直接导入散落 `.css` 文件。
  2. 样式必须采用分层目录：`tokens`、`foundations`、`features`、`utilities`，并按固定顺序加载。
  3. 历史 CSS 变量需统一收敛为可追踪设计令牌；缺失令牌必须提供兼容别名，禁止运行时出现未定义 `--vb-*`。
  4. 同一业务域样式必须保证单一来源（首批覆盖 Git：`.git-*` 仅由 `features/git` 提供）。
  5. 旧模板样式链（`apps/desktop-tauri/src/styles.css`）需同步升级到 SCSS，保持仓库样式技术栈一致。
- NFR：
  1. 视觉回归控制在“等价迁移”范围，不进行大规模视觉重设计。
  2. 迁移期间必须可分阶段发布，任何阶段都可执行构建与运行。
  3. 依赖变更满足治理约束：新增依赖需进入白名单并执行最小验证链路。
  4. 样式构建不得引入不可控全局副作用（如重复来源覆盖、未命名空间扩散）。
- 前置条件：`docs/07` 允许引入 `sass`；当前主题系统基于 `--vb-*` 令牌。
- 主流程：
  1. 新建 SCSS 样式骨架并切换主入口。
  2. 迁移现有样式文件到分层目录，保持类名与 DOM 结构稳定。
  3. 补齐兼容令牌并移除组件侧分散样式导入。
  4. 按 feature 清理重复规则并建立单一来源。
  5. 执行 `npm run typecheck`、`npm run build:web`、`cargo check --workspace` 验证。
- 异常流程：
  1. 构建失败（Sass 规则冲突）-> 回退到最近可编译状态，逐层定位入口与加载顺序问题。
  2. 视觉偏差超阈值 -> 保持语义类不变并恢复兼容 token，延后深层重构。
  3. 未定义变量导致渲染异常 -> 在 token 兼容层补齐别名，禁止在 feature 层硬编码兜底。
- 验收标准：
  1. `src/styles/index.scss` 成为唯一业务样式入口，核心组件不再直接导入本地 `.css`。
  2. Web 构建与类型检查通过，Rust 工作区检查通过。
  3. Git 相关样式去重后仍保持功能与视觉可用。
  4. 主题切换与 ambient 参数在迁移后行为一致。

### WF-22 外部通道入站路由与派发流程（T-172）

1. GT Office 启动时在 `127.0.0.1` 拉起 webhook runtime，生成随机 token 并落盘 `~/.gtoffice/channel/runtime.json`。
2. 通道入口接收外部消息，转换为 `ExternalInboundMessage`。
3. 读取或生成 `idempotency_key`，先查幂等缓存；命中直接返回重复结果。
4. 按绑定优先级执行路由：`peer > group > account > channel default > hint`。
5. 执行准入判定：
   - `open`：直接放行
   - `allowlist`：仅 allowlist 放行
   - `pairing`：allowlist 未命中时生成 pairing code 并拒绝派发
   - `disabled`：全部拒绝
6. 校验通过后转换为 `task_dispatch_batch` 请求，派发到目标 Agent。
7. 发射外部链路事件并缓存本次结果用于后续幂等命中。
8. `system.gto_doctor` 读取外部路由/准入/幂等快照，输出诊断建议。
9. 连接器命令可执行账户健康检查与 Telegram webhook 同步，结果在设置页和事件总线可见。
10. 外部入站命中 Telegram 路由后，系统必须先将消息直接写入已绑定的 live terminal session 并提交执行；禁止为了回传而额外启动第二个 headless `claude/codex` 进程替代当前 CLI 会话。
11. 当 live terminal session 有输出写入前端 xterm 后，前端必须采集已渲染可视屏幕快照并调用 `terminal.report_rendered_screen` 上报；后端回复链路只消费 rendered-screen diff 结果，不再消费 PTY 原始 chunk 作为正文事实源。

## 6. 异常与边界场景

16. 外部通道重复投递导致重复任务
