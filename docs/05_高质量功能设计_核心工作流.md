# GT Office 高质量功能设计（核心工作流）

## 1. 设计目标

本文件将需求从“能力描述”升级为“可执行设计”，用于指导研发、测试、验收三方协同。

目标：

1. 明确每个核心功能的用户价值、输入输出、状态变化。
2. 固化主流程与异常流程，避免实现偏差。
3. 为接口设计、测试用例、迭代拆分提供直接依据。

## 2. 信息架构（IA）

`T-010` 已确认采用 Agent First（工位优先）布局，不再采用传统编辑器中心模型。

### 2.2 区域职责（Agent First）

1. 顶部控制带：全局级控制与调度，不承载局部业务细节。
2. 主导航轨道：一级能力入口（工位、任务、文件、Git、Hook、通道、策略、设置）。
3. 左侧业务面板：当前一级能力的主操作面板（可切换内容，不切换布局骨架）。
4. 中央工位画布：核心执行面，显示 Agent 工位宫格窗口，每格内嵌终端命令行与会话控制。
   - 输入交互采用终端窗口原位键盘输入，回车直接执行命令。
5. 状态栏：展示系统态、工作区态、资源态、策略态。

### 2.3 扩展性约束（面向百级 Agent）

1. 工位矩阵必须虚拟化渲染，只渲染可视区。
2. 终端输出采用分级订阅：前台实时、后台降频或摘要；终端渲染使用 xterm 并通过增量流式写入，避免高频整树重渲染。
3. 工位窗口与 PTY 会话生命周期解耦，避免 UI 卡顿影响执行。
4. 所有区域数据通过统一事件总线更新，避免点对点耦合。
5. 多工作区切换时，所有面板必须按 `workspace_id` 原子切换上下文。

## 4. 关键用户故事（示例）

### US-01 打开工作区并恢复会话（EPIC-01）

- 作为开发者，我希望打开项目后自动恢复上次窗口布局和终端状态，以减少重复操作。
- 前置条件：项目存在 `.gtoffice/config.json` 或可创建。
- 主流程：
  1. 用户选择目录打开。
  2. 系统校验目录权限并加载工作区配置。
  3. 系统恢复窗口布局、文件标签、终端会话快照。
- 异常流程：
  1. 配置文件损坏 -> 提示“恢复失败”，可“忽略并重建”。
  2. 终端命令不可用 -> 恢复会话但标记失败并提示修复 shell。
- 验收标准：
  1. 恢复流程在 3 秒内可交互。
  2. 恢复失败不阻断进入工作区。

### US-02 启动 Codex 会话并注入上下文（EPIC-05）

- 作为开发者，我希望一键启动 Codex，并自动带上当前文件与 diff 上下文。
- 前置条件：已配置 Tool Profile 且有可用命令。
- 主流程：
  1. 用户在 Tool Launcher 选择 `codex` profile。
  2. 选择上下文来源（当前文件/选中目录/当前 diff）。
  3. 系统拼装命令与参数，创建受控 Terminal Session。
  4. 会话输出流实时展示并关联 Change Feed。
- 异常流程：
  1. 命令不存在 -> 引导配置 PATH 或绝对路径。
  2. 上下文过大 -> 自动裁剪并提示摘要策略。
- 验收标准：
  1. 10 秒内完成会话可见输出。
  2. Change Feed 能标记来源会话 ID。

### US-03 查看并回滚某会话导致的文件改动（EPIC-07）

- 作为开发者，我希望看到“哪个会话改了哪些文件”，并快速回滚错误改动。
- 主流程：
  1. 打开 Change Feed，按会话筛选。
  2. 查看改动摘要与 diff。
  3. 对单文件执行回滚或打开 Git 面板做更细粒度处理。
- 验收标准：
  1. 改动来源字段完整（session_id/tool_profile/task_id/time）。
  2. 回滚动作有二次确认与结果提示。

### US-04 配置工作区与全局设置（EPIC-09）

- 作为开发者，我希望在设置页统一管理主题、终端、搜索等偏好，并区分全局与工作区作用域。
- 前置条件：已打开工作区。
- 主流程：
  1. 打开 Settings，按关键词搜索目标配置项。
  2. 修改配置并选择作用域（User/Workspace）。
  3. 系统校验并持久化，相关功能即时生效。
- 异常流程：
  1. 配置类型不匹配 -> 阻止保存并提示合法值。
  2. 工作区配置文件只读 -> 回退到用户作用域并提示。
- 验收标准：
  1. 可清晰看到配置来源与覆盖关系。
  2. 修改后无需重启即可生效（热更新项）。

### US-05 自定义快捷键并处理冲突（EPIC-10）

- 作为开发者，我希望录制快捷键并立即发现冲突，避免关键命令被覆盖。
- 主流程：
  1. 打开 Keybindings 页面，搜索命令。
  2. 录制新按键组合并保存。
  3. 系统执行冲突检查并提示替代方案。
- 异常流程：
  1. 与系统保留快捷键冲突 -> 阻止保存并给出替代键位。
  2. 多命令冲突 -> 提供“解绑旧命令后应用”选项。
- 验收标准：
  1. 保存前冲突必须可见并可处理。
  2. 快捷键修改后 1 秒内生效。

### US-06 让 AI 自动调整开发环境（EPIC-11）

- 作为开发者，我希望 AI 能按我的意图自动修改允许范围内的配置，实现完整 vibe coding 闭环。
- 主流程：
  1. AI 请求读取当前设置快照（受白名单限制）。
  2. AI 生成配置补丁并提交预览。
  3. 用户确认后系统应用补丁并记录审计日志。
- 异常流程：
  1. AI 尝试修改禁用字段 -> 拒绝并返回策略错误码。
  2. 补丁校验失败 -> 不写入并返回错误详情。
- 验收标准：
  1. 每次 AI 改动都可追溯与回滚。
  2. 敏感配置不可读不可写。

### US-07 多工作区并行开发且终端自动对齐目录（EPIC-13）

- 作为开发者，我希望同时打开多个工作区，并确保在任一工作区新建终端时自动进入该工作区根目录。
- 前置条件：已打开至少两个工作区。
- 主流程：
  1. 在 Workspace Hub 中激活目标工作区。
  2. 用户点击“新建终端”或快捷键创建会话。
  3. 系统基于 `workspace_id` 解析并注入 `cwd = workspace.root`。
  4. Terminal 标题显示所属工作区标识。
- 异常流程：
  1. 请求缺失 `workspace_id` -> 直接拒绝并返回上下文错误码。
  2. 自定义 cwd 越界到工作区外 -> 拒绝执行并提示安全边界。
- 验收标准：
  1. 任意会话都可查询到 workspace_id 与 resolved cwd。
  2. 多工作区并发运行终端时输出、任务、变更归因互不串扰。

### US-08 为 Agent 配置岗位职责并执行任务（EPIC-14/19）

- 作为团队负责人，我希望定义“实现/评审/测试”等岗位职责，并让 Agent 按职责执行任务。
- 前置条件：工作区内存在可用 Charter（`.gtoffice/agents/roles/*.md`）。
- 主流程：
  1. 在 Agent Manager 中创建 Agent 并绑定岗位角色。
  2. 系统加载 Charter，生成策略快照并校验。
  3. 分配任务，Agent 在策略约束下执行。
- 异常流程：
  1. Charter 缺失或语法错误 -> Agent 不可启动。
  2. Agent 执行越权命令 -> `policy denied` 并记录审计。
- 验收标准：
  1. 每次任务执行都可关联 `agent_id/role_id/workspace_id`。
  2. Charter 变更可触发策略快照更新并可追溯版本。

### US-09 Agent 间交接与协作通信（EPIC-15）

- 作为执行 Agent，我希望通过结构化消息把阻塞和交接信息发送给其他 Agent，避免上下文丢失。
- 主流程：
  1. Agent 发送 `handover` 消息到目标通道。
  2. 接收方确认并写入任务上下文。
  3. 任务中心更新交接链路状态。
- 异常流程：
  1. 消息投递失败 -> 自动重试并告警。
  2. 重复消息 -> 去重并保留首条有效记录。
- 验收标准：
  1. 同通道消息序列连续可验。
  2. 交接消息可一键跳转关联任务与文件改动。

### US-10 通过 Hook 自动化执行质检流程（EPIC-16）

- 作为维护者，我希望在 `git commit` 后自动触发测试、审查与通知流程。
- 主流程：
  1. 配置 `git.commit.succeeded` 事件订阅。
  2. Hook 引擎按策略触发命令/任务/通知。
  3. 结果写入 Hook 运行日志并回传任务中心。
- 异常流程：
  1. Hook 超时 -> 熔断并标记失败。
  2. Hook 目标脚本异常 -> 记录错误并继续主流程。
- 验收标准：
  1. Hook 失败不阻断主业务流程。
  2. 同事件多个订阅互不影响。

### US-11 观察 Agent 信息流并定位异常（EPIC-17）

- 作为操作者，我希望在图视图看到 Agent、任务、消息、文件改动关系，快速定位异常链路。
- 主流程：
  1. 打开 Agent Graph，选择工作区与时间范围。
  2. 图中高亮失败节点与关键路径。
  3. 点击节点查看详情并执行跳转（任务/终端/diff）。
- 验收标准：
  1. 可从任意异常节点追溯上游触发源。
  2. 图数据与审计日志一致。

### US-12 顶部控制带集成到原生窗口顶部（T-065）

- FR：
  1. 顶部控制带在桌面端进入“标题栏形态”，占用更少垂直空间并保持原有按钮能力。
  2. 标题栏区域必须提供可拖拽窗口能力，且不影响输入框/按钮点击。
- NFR：
  1. 不新增后端命令与事件，保持现有契约稳定。
  2. 跨平台行为可降级：macOS 使用 overlay 标题栏；Windows/Linux 保持原生系统标题栏并复用统一 UI。
  3. Windows 无边框场景需具备“拖动中渲染降级”机制；禁止把高频 `window.onMoved` 逐帧桥接到前端作为样式触发链路，优先采用 `app-region` + 低频/静态降级策略，避免拖拽退化为逐帧移动。
- 作为开发者，我希望顶部操作按钮尽可能贴合系统窗口顶部，从而减少界面层级并提升可用空间。
- 前置条件：运行在 Tauri 桌面壳层。
- 主流程：
  1. 系统将顶部控制带渲染为标题栏样式，并保留 workspace/git/settings 等操作入口。
  2. 在标题栏内提供专用拖拽区域，用户可直接拖动窗口。
  3. macOS 启用 `titleBarStyle=Overlay`，控制带与原生窗口头部对齐。
- 异常流程：
  1. 非 Tauri 运行时（Web 预览）-> 自动回退为普通顶部栏样式。
  2. 平台不支持 overlay -> 保持默认原生标题栏，不阻断按钮可用性。
- 验收标准：
  1. 顶部按钮功能不回归（打开工作区、刷新 Git、打开设置）。
  2. 桌面端可在顶部区域拖拽窗口，按钮与输入框仍可点击。
3. 页面主内容区域可视高度较改造前提升（去除独立卡片式顶部占位）。

### US-13 工位卡片管理与沉浸全屏终端（T-067）

- FR：
  1. 中央工位画布支持运行时新增/删除工位卡片，并保持每卡片终端能力独立。
  2. 工位卡片顶部信息默认压缩展示，可手动展开查看完整终端元信息。
  3. 双击工位卡片或触控板放大手势可进入该工位全屏；支持退出按钮与缩小手势退出。
  4. 工位进入全屏时，其他工位会话持续在后台运行，不应被终止。
  5. 工位卡片需设置最小高度，宫格区域优先内部滚动，不允许通过持续压缩卡片破坏可读性。
  6. 删除工位必须先执行会话终止（kill），成功或可确认已不存在后方可移除卡片。
  7. 真终端文本、ANSI 调色与光标样式需保持高对比可读，避免暗底暗字。
- NFR：
  1. 终端输入链路不得重复发送/重复渲染字符（单次按键仅一次写入显示）。
  2. 终端输出订阅需避免重复监听（兼容 React Strict Mode 双挂载），防止性能抖动与输出翻倍。
  3. 全屏与普通态切换不应触发整页滚动模型变化，仅中央画布内部滚动。
  4. 工位卡片 hover 态不应出现白屏闪烁或重绘抖动。
  5. 未激活终端光标需保持弱化可见，不影响视觉整洁性。
  6. 终端输入需采用低延迟写入策略（空闲即发 + 拥塞批量 + 回车即时发送），避免按键交互粘滞。
  7. 工位数量较大时中央画布需启用虚拟化渲染，仅渲染可视区域卡片，避免 UI 掉帧与内存抖动。
  8. 终端高频输出导致的未读计数/徽章更新必须批量合并（短周期 flush 或 rAF），禁止按每个输出 chunk 逐条触发全局状态更新。
- 作为开发者，我希望在中央画布里动态管理工位，并在需要时快速放大某个工位专注操作，而不打断其他工位执行。
- 前置条件：工作区已绑定，至少存在一个可渲染工位卡片。
- 主流程：
  1. 用户点击“新增工位”创建新卡片，新卡片默认未启动会话并可立即交互。
  2. 用户点击删除按钮移除工位，系统终止其会话并回收前端会话映射。
  3. 用户双击工位或触控板放大手势，进入工位全屏终端视图。
  4. 用户点击“退出全屏”或触控板缩小手势，恢复宫格视图。
- 异常流程：
  1. 删除工位时会话终止失败 -> 记录错误提示，但仍允许 UI 侧移除卡片避免阻塞操作。
  2. 全屏目标工位已被删除 -> 自动退出全屏，回到普通宫格态。
- 验收标准：
  1. 新增/删除工位可用，删除后不会残留无效会话映射。
  2. 工位卡片顶部信息默认收起，占位显著减少。
  3. 双击/手势放大进入全屏，退出按钮/手势缩小可退出，全程终端可持续输入输出。
  4. 输入字符不再出现重复（例如输入 `a` 显示为单个 `a`）。
  5. 工位数量增多时，中央画布出现内部滚动，单卡最小高度和终端可见性保持稳定。
  6. 键入终端时主观延迟明显降低，回车触发命令无拖尾感。
  7. 大量工位（如 100+）场景滚动仍流畅，CPU 与内存占用无明显异常飙升。

### US-14 工作区打开性能与稳定性加固（T-070）

- FR：
  1. 打开工作区时不应被文件监听初始化阻塞，UI 需优先返回“已绑定”可交互状态。
  2. 文件目录列表与 Git 状态查询需限制单次返回规模，避免大仓库导致前后端卡顿或崩溃。
  3. 工作区打开后若监听初始化失败，必须通过事件告警而不是导致主流程失败。
- NFR：
  1. 工作区打开主流程应在大目录场景下保持可响应，避免“长时间卡住后闪退”。
  2. 大仓库下内存峰值应可控（目录列表和 Git 状态都要有硬上限）。
- 前置条件：目标目录可访问，且可能包含大量文件/未跟踪改动。
- 主流程：
  1. 用户触发 `workspace.open`。
  2. 系统完成工作区绑定后立即返回成功响应。
  3. watcher 初始化在后台异步执行，失败时发 `filesystem/watch_error`。
  4. 前端异步刷新 Git 状态与目录，不阻塞工作区可用性。
- 异常流程：
  1. watcher 初始化失败 -> 记录错误并提示，不回滚已打开的工作区。
  2. Git 或目录返回量过大 -> 启用限流策略，避免前端崩溃。
- 验收标准：
  1. 打开大仓库时 UI 可快速恢复交互，不出现明显冻结。
  2. 目录与 Git 响应有上限保护，应用稳定不闪退。
  3. watcher 异常只影响监听能力，不影响工作区打开结果。

### US-15 岗位目录与 Agent 工作区同步治理（T-072）

- FR：
  1. 每个岗位必须映射到独立目录：`.gtoffice/org/{role}`。
  2. 每个 Agent 必须映射到岗位目录下独立工作区：`.gtoffice/org/{role}/{agent_id}`。
  3. 工位总览与中央工位卡片必须展示同一份目录元数据并保持同步。
  4. 终端会话创建时必须将 `cwd` 绑定到对应 Agent 工作区目录，而非仅绑定工作区根目录。
  5. 岗位变更时，目录映射需立即同步；若已有活动终端会话，先终止并重置会话映射，避免目录漂移。
- NFR：
  1. 目录映射规则需模块化封装，避免路径拼接逻辑散落在多个组件。
  2. 终端目录绑定流程必须跨平台可用（不依赖特定 shell 的 `mkdir/cd` 命令语法）。
  3. 目录创建与终端创建总耗时应保持在可接受范围，不引入明显交互卡顿。
- 前置条件：已绑定工作区，且具备 `fs.write_file` 与 `terminal.create` 能力。
- 主流程：
  1. 创建或加载工位时，根据 `role + agent_id` 计算岗位目录和 Agent 工作区目录。
  2. 在启动终端前先通过文件系统命令确保 Agent 工作区目录存在。
  3. 终端使用 `cwdMode=custom` 创建，并将 `cwd` 指向 Agent 工作区绝对路径。
  4. UI 多视图（总览/画布）统一展示目录信息。
- 异常流程：
  1. 读取 `workspace.get_context` 失败 -> 阻断会话创建并提示上下文不可用。
  2. 目录创建失败 -> 阻断会话创建并提示失败原因，不进入错误 cwd。
  3. 岗位切换时终端 kill 失败 -> 保留原会话并提示错误，避免误清映射导致孤儿进程。
- 验收标准：
  1. 不同岗位对应不同目录，不同 Agent 在同岗位下目录互不重叠。
  2. 新建终端后 `resolvedCwd` 与 Agent 工作区目录一致。
  3. 岗位变更后，目录字段在总览和画布实时一致。
  4. 目录创建/上下文失败时有明确错误反馈，不会创建到错误目录。

### US-16 任务中心发布与 Agent 接收反馈（T-074）

- FR：
  1. 人类用户可在任务中心使用 Markdown 编辑任务内容，并支持常见模板（小节/代码块/清单）快速插入。
  2. 附件字段在协议层保留兼容；M2.1（`T-147`）起前端不再提供附件输入入口。
  3. 发布任务时必须可指定目标 Agent（工位），并将任务内容落盘到 `.gtoffice/tasks/{task_id}/task.md`。
  4. 任务发布后需在目标工位终端写入派发提示，并在中央画布对应工位卡片展示“已接收”气泡反馈。
  5. “已接收”气泡必须自动消退（约 3 秒），不影响终端输入操作。
- NFR：
  1. 任务中心与工位卡片联动只更新目标工位，避免全画布高频重渲染。
  2. 任务发布链路复用现有 filesystem/terminal 能力，不新增重型依赖。
  3. 任务派发历史需可追溯（发送中/成功/失败）并保留失败详情。
- 前置条件：已绑定工作区；至少存在一个可用工位。
- 主流程：
  1. 用户在任务中心输入标题与 Markdown 内容，选择目标 Agent。
  2. 系统生成 `task_id`，将 `task.md` 与 `manifest.json` 写入 `.gtoffice/tasks/{task_id}/`。
  3. 系统确保目标工位终端会话存在并发送派发命令提示。
  4. 中央工位卡片展示“任务已收到”气泡约 3 秒并自动消退。
- 异常流程：
  1. 未绑定工作区 -> 阻断发送并提示先绑定工作区。
  2. 任务内容为空 -> 阻断发送并提示补充 Markdown 内容。
  3. 目标工位会话创建失败 -> 派发记录标记失败并保留错误详情。
- 验收标准：
  1. 任务中心可完成 Markdown 编辑与模板插入。
  2. 任务成功发送后目标工位出现 3 秒接收气泡反馈，并包含任务标识。
  3. 派发记录正确呈现 `sending/sent/failed` 状态，失败时可看到错误详情。
  4. 任务文件按约定目录落盘，且终端可看到派发提示日志。

### US-17 任务中心可靠性增强（T-076）

- FR：
  1. 任务中心必须支持对 `failed` 状态任务执行一键重发，目标 Agent 与任务路径保持一致。
  2. 重发前需校验任务文件可读，避免将无效路径投递到终端。
  3. 任务草稿与派发历史需按 `workspace_id` 持久化，支持切换/重启后恢复。
  4. 持久化采用双通道：浏览器本地缓存 + 工作区文件 `.gtoffice/tasks/.task-center-draft.json`。
- NFR：
  1. 持久化写入需防抖，避免高频输入导致频繁 I/O。
  2. 加载持久化数据时不得阻塞主界面交互；读取失败需静默降级到内存默认值。
  3. 重发链路复用现有终端派发逻辑，不新增重复实现。
- 前置条件：任务中心已有历史记录，且至少一条任务状态为 `failed`。
- 主流程：
  1. 用户点击失败任务“重发”按钮。
  2. 系统校验任务文件存在并确保目标工位会话可用。
  3. 系统重发任务指令并更新状态为 `sent`。
  4. 草稿与历史状态按防抖策略持久化到双通道存储。
- 异常流程：
  1. 任务文件不存在 -> 重发失败并记录 `detail`。
  2. 目标工位不存在或会话创建失败 -> 重发失败并提示。
  3. 文件持久化失败 -> 保留 localStorage 快照并继续可用。
- 验收标准：
  1. 失败任务可一键重发并在成功后转为 `sent`。
  2. 关闭并重新打开应用后，任务草稿与派发历史可恢复。
  3. 工作区切换时，任务中心数据按工作区隔离，不串数据。

### US-23 任务中心多选派发与 Agent 通道（T-146）

- FR：
  1. 任务中心支持多选目标 Agent（工位），一次发布可派发到多个目标。
  2. 一次批量发布生成多个独立 `task_id`（每个目标一个），状态可独立追踪与重试。
  3. 平台提供统一 Agent 通道抽象，支持 `task_instruction/status/handover` 三类消息，供任务中心与后续 Agent->Agent 互发复用。
  4. 通道本期仅做传输不持久化：在线目标必须返回投递 ACK；离线目标立即失败并回传失败原因。
  5. 目标 Agent 收到消息后，中央工位卡片必须出现“任务已收到”反馈气泡（约 3 秒自动消退）。
- NFR：
  1. 批量派发过程中禁止全局高频重渲染，只更新目标工位与任务中心历史区。
  2. 通道序列号在同通道内单调递增，便于后续乱序排查与观测。
  3. 重逻辑在 Rust 侧编排，前端仅做输入、渲染与事件订阅。
- 前置条件：已绑定工作区；目标工位已在运行态（或可按现有链路自动拉起会话）。
- 主流程：
  1. 用户在任务中心填写任务内容并多选目标 Agent。
  2. 系统批量确保目标 Agent runtime 在线，随后调用 `task_dispatch_batch`。
  3. Rust 侧为每个目标生成独立 `task_id`，写入 `.gtoffice/tasks/{task_id}/`，并向目标 session 写入派发命令文本。
  4. 前端在命令写入成功后通过 xterm 原生输入链路触发 submit（`terminal.input('\r', true)`），再接收 `channel/message` 与 `channel/ack` 事件更新工位反馈与派发历史状态。
- 异常流程：
  1. 目标 Agent 离线 -> 返回 `failed`，详情包含 `AGENT_OFFLINE`。
  2. 任务文件写入失败 -> 返回 `failed`，详情包含 `TASK_PERSIST_FAILED`。
  3. 通道发送失败（terminal write rejected）-> 返回 `failed`，详情包含 `CHANNEL_DELIVERY_FAILED`。
- 验收标准：
  1. 多选 N 个目标发布后，历史中可看到 N 条独立记录（含 `batch_id + task_id`）。
  2. 在线目标显示 `sent`，离线目标显示 `failed` 且保留错误详情。
  3. 目标工位出现“任务已收到”气泡，非目标工位不出现误提示。
  4. 前端重启后任务草稿仍可恢复，通道消息不要求恢复（本期无持久化）。

### US-18 区域间动态氛围灯与设置开关（T-096）

- 前置条件：主界面已完成 Top/Main/Status 三段布局，并具备设置页展示能力。
- 主流程：
  1. 用户进入主界面，系统按当前主题与强度档位渲染底层多彩氛围灯背景。
  2. 用户在设置页切换“氛围灯”开关。
  3. 用户在设置页切换强度档位（弱/中/强）。
  4. 系统立即切换显示状态并持久化偏好。
- 异常流程：
  1. 布局尺寸变化（窗口缩放/侧栏收起）-> 底层背景保持连续动画，无需缝隙重算。
  2. 低性能或减少动态偏好场景 -> 动画降级但不影响主流程交互。
- 验收标准：
  1. 默认开启时，整屏背景可见连续多彩流动光效。
  2. 浅色/深色主题的氛围灯视觉明显不同且风格一致。
  3. 设置开关关闭后光效完全消失，开启后即时恢复。
  4. 设置强度切换到弱/中/强后，光效强度有明确层级差异。
5. 高频交互（终端输入、滚动、拖拽）无明显卡顿回归。

### US-22 Sidecar Daemon 高性能核心链路（T-098）

- FR：
  1. Tauri 主进程只保留壳层与权限，不承载文件扫描、全文搜索、PTY 生命周期等重逻辑。
  2. Rust Core Daemon 通过本机 socket 与 UI 通信，协议使用二进制帧（长度前缀 + bincode）。
  3. 搜索必须支持：新请求取消旧请求（cancel token）、分批流式返回（chunk events）、前端背压反馈（降采样/暂停推送）。
  4. 文件树目录读取必须支持懒加载与分页 cursor，禁止全量递归扫描阻塞 UI。
  5. 终端会话必须支持大量并发创建，输出链路需具备 ring buffer 缓冲以应对消费抖动。
- NFR：
  1. socket writer 必须有有界队列，形成统一背压边界。
  2. 所有阻塞/CPU 密集任务必须在 tokio blocking pool 或独立线程执行，不能阻塞 async reactor。
  3. 多平台行为一致：Windows/macOS/Linux 均可运行；平台差异收敛到 transport/terminal provider 层。
- 前置条件：已存在 workspace root，且 UI 已实现虚拟列表和增量渲染能力。
- 主流程：
  1. UI 启动时由 Tauri 拉起 sidecar daemon，并建立 socket 连接。
  2. 用户展开目录时调用 `ListDir`，按 `cursor` 增量请求下一页。
  3. 用户输入搜索词时发起 `SearchStart`，daemon 流式返回 `SearchChunk`。
  4. 用户继续输入新关键词时触发 `SearchCancel + SearchStart`，旧任务被取消。
  5. 用户创建终端时发起 `TerminalCreate`，后续通过 `TerminalOutput` 事件消费输出。
- 异常流程：
  1. socket 断开 -> UI 自动进入重连态并停止发送新请求。
  2. 搜索输出队列拥塞 -> daemon 发 `SearchBackpressure`，UI 转入降频渲染。
  3. 终端 session 异常退出 -> daemon 发 `TerminalState(exited|failed)`，UI 清理会话状态。
- 验收标准：
  1. `SearchStart` 可稳定中断旧任务并返回新流。
  2. `ListDir` 在大目录场景下保持可交互（分页 + 懒加载）。
  3. 终端输出持续流式可见，拥塞时仍不导致 UI 卡死。

## 5. 核心工作流设计

### WF-01 Workspace 打开流程

1. UI 触发 `workspace.open`。
2. `vb-security` 校验访问权限。
3. `vb-workspace` 加载配置与最近会话。
4. `vb-filesystem` 初始化目录缓存与监听。
5. `vb-git` 初始化仓库状态监听。
6. 广播 `workspace/updated` 事件。

失败补偿：

1. Git 初始化失败不阻断工作区打开，状态面板降级为“不可用”。
2. 文件监听失败时，保留手动刷新入口。

监听驱动局部刷新要求：

1. 文件监听事件统一发射 `filesystem/changed`，按 `workspace_id` 隔离。
2. 文件树仅刷新受影响父目录，不执行全树重载。
3. 删除/重命名事件必须先做本地缓存裁剪，再触发局部目录重载。
4. watcher 默认忽略 `.git/`、`node_modules/`、`target/`、`dist/` 等噪声目录与临时文件后缀，避免事件风暴。

### WF-02 Terminal 会话生命周期

状态机：

- `created` -> `starting` -> `running` -> `exiting` -> `closed`
- 异常状态：`failed`, `killed`, `timeout`

状态转换约束：

1. `starting` 超过阈值进入 `failed`。
2. `running` 可被用户中断进入 `killed`。
3. `exiting` 超时触发强制终止并记录日志。

### WF-03 Git 提交流程

1. 刷新 status -> 展示 staged/unstaged。
2. 用户选择 hunks 或文件暂存。
3. 输入 commit message（支持模板）。
4. 执行提交并刷新 log。
5. 生成 Change Feed 事件（type: `git_commit`）。

失败补偿：

1. hook 失败时展示原始输出并保留 message。
2. 索引冲突时引导用户执行冲突解决流程。

### WF-03A Git 完整协作流程（T-104）

1. 用户可在 Git 面板执行常用链路：`stage/unstage/discard/commit/log`。
2. 分支管理支持：`list_branches/checkout/create_branch/delete_branch`。
3. 远程同步支持：`fetch/pull/push`，并支持 `set-upstream` 与 `force-with-lease`。
4. 临时工作区切换支持：`stash_push/stash_pop/stash_list`。
5. 所有写操作成功后刷新 `git.status` 并广播 `git/updated`，保持多面板状态一致。

失败补偿：

1. 路径越界或参数非法直接拒绝，并返回稳定 `GIT_*` 错误码。
2. 远程操作失败时保留本地状态，不自动执行高风险补偿动作。
3. stash/pop 冲突时返回原始错误并保留当前工作树，要求用户手动解决。

### WF-03B Git 高性能面板交互流程（T-105）

1. 左侧 Git 文件变更列表采用虚拟化渲染，仅绘制可视区与有限 overscan。
2. 首次进入 `Git` 导航时并行拉取 `status/log/branches/stash`，后续写操作走局部刷新。
3. `diff_file` 仅按选中路径懒加载，切换选中项时中止旧结果落盘（按请求序号丢弃过期响应）。
4. 所有 Git 写操作（stage/unstage/discard/commit/checkout/stash）完成后统一触发：
   - `git.status` 刷新 UI 主状态
   - `git/updated` 事件驱动多面板同步
   - 轻量补拉 `log/branches/stash`
5. 事件风暴控制：`git/updated` 前端按 `96ms` 窗口合并刷新，避免频繁 IPC 导致主线程抖动。
6. 视图分工固定：左侧第二列仅承载 Git 操作（stage/commit/branch/stash 等），右侧主区承载提交历史、Git Graph 与文件 diff 详情；点击左侧变更文件后右侧立即切换对应 diff。
7. 超长历史采用分页加载：`git.log` 以 `limit + skip` 拉取历史页，默认先载入最新页；用户点击“加载更早提交”时追加旧页，“回到最新”时重置到首屏页。
8. 在 Git Graph 视图点击提交时，前端调用 `git.commit_detail` 获取该提交的作者、摘要、正文与变更文件列表，并在图谱下方展开详情区。

验收标准：

1. 2000+ 变更文件场景下，列表滚动与选中交互持续流畅，无明显掉帧。
2. Git 面板内可完成高频链路：`stage/unstage/discard/commit/branch/stash/fetch/pull/push`。
3. 多入口触发 Git 变更（顶部按钮、面板按钮、外部命令）后，状态可在一次节流窗口内自动收敛一致。
4. Git Graph 点击任意提交后，可在 300ms 内看到详情加载反馈，成功后展示变更文件列表，失败时显示结构化错误提示。

### WF-04 Tool Adapter 启动流程

1. 读取 profile（cmd/args/env/policy）。
2. 校验策略（目录白名单、网络策略、危险命令策略）。
3. 注入上下文（文件列表、diff 摘要、工作区变量）。
4. 创建终端会话并绑定 `tool_session_id`。
5. 推送任务中心与 Change Feed。

### WF-05 设置加载与合并流程（EPIC-09）

1. 读取默认配置。
2. 读取用户配置（系统目录）。
3. 读取工作区配置（`.gtoffice/config.json`）。
4. 合并会话临时覆盖项。
5. 产出 `effective_settings` 并广播 `settings/updated`。
6. 运行时消费者（filesystem watcher / 文件预览限制）按 `effective_settings` 热更新。

失败补偿：

1. 某层配置解析失败时忽略该层并提示修复。
2. 配置迁移失败时保留旧文件并生成兼容副本。

### WF-06 快捷键写入流程（EPIC-10）

1. UI 录制键位并构建 binding 变更请求。
2. `vb-keymap` 执行冲突检测与平台合法性检测。
3. 保存到用户或工作区作用域。
4. 广播 `keymap/updated` 并热更新命令路由。

### WF-07 AI 配置改动流程（EPIC-11）

1. AI 发起 `preview_patch` 请求。
2. `vb-ai-config` 校验字段白名单与策略。
3. `vb-settings` 进行 schema 校验并生成 diff 预览。
4. 用户确认后应用补丁。
5. 写入 `ai_config_audit_logs`，广播 `ai_config/changed`。

### WF-08 多工作区终端创建流程（EPIC-13）

1. UI 发起 `terminal.create` 并带 `workspace_id`。
2. `vb-workspace` 解析 `WorkspaceContext`（root、permissions、active state）。
3. `vb-terminal` 按策略解析 cwd：
   - 默认：`cwd = workspace.root`
   - 自定义：需通过路径越界校验
4. 创建 PTY 会话并写入 `terminal_sessions(workspace_id, cwd, ...)`。
5. 广播 `terminal/state_changed` 与 `workspace/active_changed`（必要时）。

### WF-09 Agent 启动与策略快照流程（EPIC-14/19）

1. UI 发起 `agent.create`，输入 `role_id` 与 `workspace_id`。
2. `vb-policy` 读取 Charter 文档并做 schema/语义校验。
3. 生成 `policy_snapshot_id` 并绑定 Agent。
4. `vb-agent` 将 Agent 状态置为 `ready`，可接收任务。

失败补偿：

1. Charter 校验失败 -> Agent 置为 `blocked` 并附错误详情。
2. 策略快照入库失败 -> 回滚 Agent 创建。

### WF-10 Channel 消息投递流程（EPIC-15）

1. Agent 发送消息到 `channel.publish`。
2. `vb-channel` 分配序列号并写入持久层。
3. 实时分发给订阅者（本地事件总线，未来可加 Redis Pub/Sub）。
4. 接收方 ACK，消息状态更新为 `delivered`。

失败补偿：

1. ACK 超时 -> 按重试策略重投。
2. 重复消息 -> 以幂等键去重。

### WF-10A 文件预览保护流程（EPIC-02）

1. UI 请求 `fs.read_file` 后，后端只读取预览窗口字节（默认 256KB）。
2. 若检测为二进制（NUL/control 比例超阈值），返回 `previewable=false`，UI 给出提示不渲染正文。
3. 若文本超过预览阈值，返回 `truncated=true` 与 `previewBytes/sizeBytes`，UI 展示截断提示并渲染部分内容。
4. 用户点击“加载完整内容”后调用 `fs.read_file_full`（受读取上限保护）。
5. 文件变更事件命中当前预览文件时，UI 自动刷新或在删除后清空预览状态。

### WF-11 Hook 执行流程（EPIC-16）

1. 业务事件进入 Hook 总线。
2. `vb-hook` 根据订阅规则匹配触发器。
3. 每个订阅生成独立 `hook_run` 任务。
4. 任务执行并写回结果，推送 `hook/executed`。

失败补偿：

1. 连续失败超过阈值 -> 自动熔断一段时间。
2. 单订阅失败不影响其他订阅。

### WF-12 Agent Graph 聚合流程（EPIC-17）

1. `vb-observability` 订阅 task/channel/hook/changefeed 事件。
2. 组装关系边（Agent->Task、Task->File、Message->Task）。
3. 写入图索引并输出查询 API。
4. 前端按过滤条件拉取并渲染图视图/时间线。

### WF-13 原生标题栏集成流程（T-065）

1. Tauri 窗口配置启用 `titleBarStyle=Overlay`（支持平台生效）。
2. WebUI 顶部控制带切换到标题栏样式（去卡片化、贴顶显示）。
3. 在控制带内声明专用 `data-tauri-drag-region`，提供拖拽区。
4. 平台适配：
   - macOS：为交通灯区域预留左侧内边距。
   - Linux：当 overlay 不可用时走无装饰窗口 + 顶部自定义窗口控制（最小化/最大化/关闭）降级路径，并保持拖拽区可用。
   - 移动端断点：自动折叠拖拽区。
5. 性能防回归：
   - Windows 拖拽区尽量保持“纯净区域”（不承载复杂玻璃/阴影/动画控件）。
   - 禁止在拖拽主链路订阅并处理高频窗口移动事件（例如 `window.onMoved -> class toggle`），避免事件桥接本身造成卡顿。
   - Windows 原生标题栏模式默认启用轻量渲染（去玻璃/大阴影/全局复杂过渡）；`dragging class` 仅作为补充降级，不作为唯一性能保障。
   - 使用 `dragging class` 的降级样式时，拖动结束后必须自动恢复，避免残留状态。
   - 终端高频输出场景下，未读数等非关键反馈需合批提交，避免每个 chunk 触发 `ShellRoot` 级重渲染影响拖拽帧率。
6. 保持所有操作按钮原有事件绑定，不引入新命令契约。

### WF-14 氛围灯渲染与开关流程（T-096）

1. UI 启动后读取显示偏好与生效设置，解析 `ui.ambientLighting.enabled`（默认 `true`）。
2. 渲染固定底层氛围灯组件（少量渐变层），并由半透明主界面容器承接光效。
3. 主题/强度切换时更新氛围灯变量（颜色、透明度、运动幅度、扰动强度）。
4. 用户在设置页切换开关或强度后，实时更新效果并持久化。
5. 布局变化（窗口缩放、左侧面板显隐）仅触发布局重排，不触发灯效测量逻辑。

### WF-15 Sidecar Daemon 请求处理流程（T-098）

1. Tauri 壳层拉起 daemon 并建立本机 socket 连接。
2. UI 请求以二进制帧写入 socket，daemon reader loop 解帧并路由到对应 service。
3. service 返回 response（一次性）或 event（流式），writer loop 统一出站。
4. 当 writer 队列拥塞时：
   - 搜索：触发 chunk 降采样并发 `SearchBackpressure`。
   - 终端：输出写入 ring buffer，等待可写后再 flush。
5. client 断开时，daemon 回收该 client 关联的 search task 与 terminal sessions。

### WF-16 中央工位滚动性能优化流程（T-099）

1. 中央工位列表在达到阈值后启用 `@tanstack/react-virtual`，并使用低阻塞配置（`useFlushSync=false`、`useScrollendEvent=true`、`useAnimationFrameWithResizeObserver=true`）。
2. 工位卡片链路保证回调引用稳定，避免滚动过程中因 props identity 抖动触发批量重渲染。
3. 卡片组件使用显式 `memo` 比较器，仅在可见内容（激活态、预览文本、未读数、全屏态）变化时重渲染。
4. 移除会阻塞合成线程的 `wheel passive:false` 监听路径，保留按钮/双击/手势作为全屏入口。
5. 滚动容器启用 `contain`、`overscroll-behavior`、`scrollbar-gutter` 与定制滚动条，兼顾滚动稳定性和视觉一致性。

### WF-17 全局滚动同类实现一致性修复流程（T-100）

1. 扫描 WebUI 中所有高频滚动容器（侧栏、文件树、历史列表、弹窗列表）与滚动事件绑定点。
2. 对 `onScroll -> setState` 类链路改为 `requestAnimationFrame` 节流提交，避免高频滚动触发同步重渲染。
3. 对主要 `overflow:auto` 容器统一应用低阻塞滚动策略：`overscroll-behavior`、`scrollbar-gutter`、`contain`、统一滚动条样式。
4. 保留必要交互语义（键盘、焦点、选择）不变，确保仅优化滚动性能与视觉反馈。
5. 使用构建与类型校验做回归，保证样式和交互链路无破坏性变化。

### WF-18 大列表极速滚动空白修复流程（T-101）

1. 将文件树手写虚拟滚动替换为 `@tanstack/react-virtual`，由库管理高速滚动窗口计算与渲染同步。
2. 提高虚拟列表 `overscan`，确保快速滚动时下一屏内容提前就绪，避免可视区短暂空白。
3. 开启 `useScrollendEvent` 与 `useAnimationFrameWithResizeObserver`，减少滚动/尺寸变化的更新抖动。
4. 滚动进行中禁用目录展开入场动画，避免动画开销与虚拟列表滚动竞争主线程。
5. 对大列表执行快速滚动回归，验证“无白屏、无延迟回填”。

### WF-19 文件搜索弹窗与高性能检索流程（T-102）

1. 文件管理面板只保留“搜索入口”按钮，点击或快捷键拉起搜索弹窗（同设置弹窗视觉语义）。
2. 搜索弹窗支持“文件名 / 内容”模式切换，输入时按模式触发对应检索链路。
3. 内容搜索后端采用并行扫描并遵循标准忽略规则（隐藏目录默认不参与），减少无效 I/O。
4. 同一工作区下新请求自动 supersede 旧请求，旧任务尽快停止，避免连续输入造成资源堆积。
5. 结果按列表增量展示并支持 Enter 命中首项跳转，关闭弹窗不影响主界面交互帧率。

### WF-20 文件编辑器快捷键路由与内搜流程（T-148）

1. Shell 全局快捷键在捕获阶段先判断上下文：编辑器焦点、普通输入框焦点、当前导航页。
2. 当焦点在文件编辑器（或文件页且存在活动文件）时，`Ctrl/Cmd+F` 路由到 CodeMirror `openSearchPanel`，不触发文件搜索弹窗。
3. `Ctrl/Cmd+H` 路由到编辑器替换入口；`F3/Shift+F3` 路由到查找下一项/上一项。
4. `Ctrl/Cmd+Shift+F` 统一保留为工作区全文搜索，触发文件搜索弹窗 content 模式。
5. `Ctrl/Cmd+P` 统一作为文件名/路径搜索入口；`Ctrl/Cmd+F` 在非文件编辑上下文不再回退全局搜索。
6. 全局快捷键命令路由由可配置映射驱动（默认值 + `keybindings.overrides`），避免 `key === 'x'` 写死分支。
7. 编辑器搜索面板样式统一接入系统设计令牌，确保主题一致与可访问焦点反馈。

## 6. 异常与边界场景

1. 超大仓库（>100k 文件）
   - 文件树按层懒加载，禁止全量展开。
2. CLI 输出洪峰
   - PTY 输出分块 + 背压控制，防止 UI 卡顿。
3. 外部程序修改文件
   - `notify` 事件去抖 + 增量刷新。
4. 多窗口同时操作同一仓库
   - 写操作加工作区级互斥提示，避免竞态误覆盖。
5. 系统 git 与 git2 结果不一致
   - UI 标记“降级路径”，并记录诊断日志。
6. 配置层级冲突导致结果不可预期
   - 设置面板展示“生效值 + 来源层级”。
7. 快捷键与系统保留键冲突
   - 阻止保存并给替代方案。
8. AI 连续错误补丁
   - 启用节流与失败熔断，避免污染配置。
9. 多工作区切换时终端错绑目录
   - 终端创建必须强制 workspace_id，并校验 cwd 落在 workspace 范围内。
10. Agent 通道消息乱序或丢失
   - 通道内序列号 + ACK 重试 + 持久化补偿。
11. Hook 风暴导致系统过载
   - 并发上限、限流、熔断、优先级队列。
12. Charter 被误改造成大面积越权
   - 变更评审 + 版本回滚 + 生效前 dry-run。
13. 岗位切换后终端仍停留旧目录
   - 岗位变更时先 kill 会话再重置映射，下一次会话按新目录创建。
14. 工位滚动时出现滚动条延迟与掉帧
   - 移除阻塞滚动监听，收敛渲染更新面，并将虚拟化更新切换到非 `flushSync` 模式。
15. 全局快捷键与编辑器快捷键冲突
   - 采用上下文感知路由（编辑器优先、输入框豁免、全局搜索保留），禁止“一刀切”全局抢占。

## 8. 实施拆分建议（开发顺序）

1. 先打通 EPIC-01/13/02/03 最小闭环（多工作区 + 能跑终端 + 能看文件）。
2. 接入 EPIC-05 Tool Adapter，形成 AI CLI 主路径。
3. 补 EPIC-04 Git 与 EPIC-07 Change Feed，提升可控性。
4. 加入 EPIC-09/10 设置与快捷键，打通个性化能力。
5. 接入 EPIC-11 AI 配置通道并补全审计回滚。
6. 最后强化 EPIC-08 与 EPIC-12 的安全和设计一致性。
7. 进入 EPIC-14/19，完成 Agent 岗位治理与 Charter 策略化。
8. 接入 EPIC-15/16，打通通信通道与 Hook 自动化。
9. 最后引入 EPIC-17 可视化；EPIC-18（Redis）延期到 V3+。

## 9. 设计评审门槛（进入开发前）

每个功能点进入编码前需满足：

1. 有明确 User Story 与验收标准。
2. 有主流程和至少 2 个异常流程。
3. 有接口入参与错误码定义。
4. 有最少 1 条自动化或可复现验证步骤。
