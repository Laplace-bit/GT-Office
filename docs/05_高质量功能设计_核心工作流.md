## 4. User Stories（活跃）

### US-26 多 Agent MCP 桥接通信（T-171）

- FR：
  1. manager 可派发任务到一个或多个执行 agent。
  2. 执行 agent 可回传 `status/handover` 给 manager。
  3. 支持 Codex/Claude/Gemini/Qwen 的 MCP 配置自动安装。
  4. Agent 供应商页面进入时必须先检测本机 CLI 是否已安装；已安装项隐藏安装入口，仅对未安装项展示安装 CTA。
- NFR：
  1. 桥接仅监听 `127.0.0.1` 且 token 鉴权。
  2. 请求超时与重试策略稳定，错误码可机读。
  3. 安装流程幂等，重复执行不产生重复配置。
  4. 支持“GT Office 桌面端运行于 Windows、agent/MCP 运行于 WSL”的混合部署；发送链路必须优先选取与目标 workspace 同源的 runtime/directory 状态根。
- 前置条件：GT Office 已启动；目标 `workspace_id` 有可用 runtime。
- 主流程：
  1. 用户打开 Agent 供应商页面时，前端并行查询各 CLI 的安装状态与前置环境状态。
  2. 已安装项展示“已安装”状态并隐藏安装按钮；未安装项根据环境状态展示显著安装按钮或前置依赖提示。
  3. 协作前先调用 `gto_get_agent_directory`，拿到当前 workspace 的角色、agent、在线 runtime 与 `self` 身份；若未显式传 `workspace_id`，系统可从 env 或最近 snapshot 自动推断。
  4. manager 按目录结果中的 `agent_id` 调用 `gto_dispatch_task`；该动作会将文本写入目标 agent terminal，并按 runtime 的 `submitSequence` 自动提交执行。
  5. 系统派发并返回 `batchId + per-target status`。
  6. 执行 agent 默认从当前 terminal env 自动识别 `sender_agent_id`，再调用 `gto_report_status/gto_handover` 回传；这两类 tool 仅写入协作消息，不负责向目标 terminal 注入命令。
- 异常流程：
  1. 桥接不可达 -> `MCP_BRIDGE_UNAVAILABLE`。
  2. 鉴权失败 -> `MCP_BRIDGE_AUTH_FAILED`。
  3. 目标离线 -> `AGENT_OFFLINE`。
  4. bridge 不可达但本地 `directory.json` 存在 -> 目录查询仍返回最近快照，health 返回降级诊断。
- 验收标准：
  1. 至少一个 CLI 成功完成“目录查询 -> 派发 -> handover 回传”闭环。
  2. 一发多目标返回独立结果且与任务中心一致。
  3. 未显式传 `sender_agent_id` 时，`handover/status` 仍可通过当前 agent terminal env 成功发送。
  4. 页面进入后已安装 CLI 不再显示安装按钮，未安装项安装按钮在亮暗主题下都具备清晰可见的 CTA 对比度。

### US-27 外部通道接入与多 Agent 路由闭环（T-172）

- FR：
  1. 支持 `feishu/telegram` connector 账号配置、健康检查与 route 绑定。
  2. 支持 `feishu/telegram` webhook 入站；其中 Feishu 需额外支持桌面端长连接模式配置与探测。
  2. 入站消息必须走统一路由与准入策略（`pairing/allowlist/open/disabled`）。
  3. 必须有幂等去重与统一外部链路事件。
  4. 目标 `role:<role_key|role_id>` 可解析为可投递 agent 集合。
  5. 回复链路必须采用“双源观察”：
     - `RenderedScreenSnapshot` 负责 preview、ready/finalize 边界与交互 prompt。
     - provider `session log` 负责高置信最终正文，当前 v1 支持 Claude/Codex。
     - session log 不可用时，自动回退到 rendered-screen，再回退到 VT 文本。
- NFR：
  1. 外部链路失败不影响主任务中心与终端主路径。
  2. 幂等命中 O(1) 查询。
  3. 渲染快照上报与 terminal 显示解耦，失败不阻塞执行。
  4. Windows 生产包下，connector polling / health probe / outbound send 等后台子进程不得弹出系统控制台窗口。
  5. 非 Windows 平台保持现有子进程行为，不因该修复引入新的启动副作用。
- 前置条件：已配置 channel 绑定；目标 workspace/runtime 在线。
- 主流程：
  1. 用户在 GT Office 内选择 channel、配置 connector 账号，并在平台侧完成 Bot / 权限 / 事件订阅。
  2. 系统执行 connector health probe，并返回 bot identity、callback 匹配状态与 runtime 状态。
  3. 用户保存 route binding 与 access policy。
  4. webhook 或长连接入站归一化为 `ExternalInboundMessage`。
  5. 路由匹配 + 准入校验 + 幂等判重。
  6. 写入已绑定 live terminal session 并执行。
  7. 前端持续上报 rendered-screen；后端基于 rendered-screen 生成 preview、interaction prompt 与 finalize 边界。
  8. 后端按 runtime 的 `tool_kind + resolved_cwd + injected_input` 绑定 provider session log；Claude/Codex 轮询 session log 作为 finalize 正文主源。
  9. Telegram 交互 prompt 需要支持两类回流：
     - 语义按钮：`gto:<submit_text>`
     - 方向键按钮：`gto-key:up|down|enter|esc|tab`
- 异常流程：
  1. route miss -> `CHANNEL_ROUTE_NOT_FOUND`。
  2. pairing 未通过 -> `CHANNEL_PAIRING_REQUIRED`。
  3. connector health 未通过 -> 不允许完成最终绑定。
  4. session log 缺失、歧义或 provider 未支持 -> 自动回退到 rendered-screen / VT fallback，不额外拉起 headless CLI。
  5. rendered-screen 置信度不足且无 session log 正文 -> 正文静默，不发送噪声回复。
- 验收标准：
  1. 双通道均可完成“connector 配置 -> health probe -> route 绑定”闭环。
  2. 双通道入站均可进入统一派发链路。
  3. 同一幂等键重放 3 次只派发 1 次。
  4. 权限确认/菜单内容不混入正文；Telegram 语义按钮与 `gto-key:*` 方向键回调都必须回写同一 live session。
  5. Claude/Codex 在 session log 可用时，final body 优先来自 session log；preview 仍来自 rendered-screen。
  6. Feishu v1 仅展示交互提示文本，不提供远程方向控件。
  7. Windows 安装包在存在 Telegram polling 账号时启动后不再反复弹出 `cmd`/控制台窗口。
  8. Linux/macOS 开发与生产环境下 connector 健康检查、发送与轮询行为保持不变。

### US-30 Agent 供应商配置中心（T-171）

- FR：
  1. `ai` 设置页必须升级为统一的 Agent 配置中心，而不是仅展示安装卡片。
  2. Claude Code 必须支持供应商切换、模型覆盖、API Key 托管，并强制走“预览 -> 校验 -> 确认 -> 应用 -> 审计”。
  3. Codex CLI 与 Gemini CLI 在 v1 只提供安装状态、官方文档、本地配置路径和轻量说明，不暴露复杂供应商表单。
  4. Claude 供应商必须内置精选预设与新手引导，覆盖充值入口、API Key 获取入口和推荐模型。
  5. Claude API Key 只能保存到系统凭据库；`.gtoffice/config.json` 只允许保存 `secretRef` 与非敏感配置。
  6. Claude/Codex/Gemini 配置向导中的“无输入纯过渡步骤”必须自动跳过或默认落在首个可操作步骤，禁止用户为继续流程额外点击一次“下一步”。
  7. Claude 每次成功应用后的供应商配置都必须保存到数据库，并在 Agent 供应商列表中的独立“已保存供应商”子入口里可查看、可直接切换；“配置向导”和“已保存供应商”禁止做成同一弹窗内 tab。
  8. Claude 选择 `official` 时，必须把当前生效配置切换为官方档案，并用该官方档案覆盖此前生效的第三方供应商状态；预览、快照与已保存记录中必须回填 Anthropic 官方基线 `provider/baseUrl/model/authScheme`，不能继续显示上一个第三方供应商残留值。
- NFR：
  1. 设置页在 `375/768/1024/1440` 下无横向滚动。
  2. Claude 预设切换与预览反馈必须在 1 个面板内完成，避免用户在多个弹窗之间来回跳转。
  3. 相同逻辑必须抽成 shared 组件；Claude / Codex / Gemini 目录必须分开。
  4. Claude 配置应用后必须同步受管环境变量到系统级 `~/.claude/settings.json`，并保留已有 `mcpServers`、权限及非受管 env 字段；不要求热切换已运行会话。
- 前置条件：已打开 workspace；若配置 Claude 供应商，则本机可安装或已安装 Claude CLI。
- 主流程：
  1. 用户进入 Agent 配置中心，前端读取 `ai_config.read_snapshot`，同时拿到 3 个 Agent 的安装状态、Claude 当前配置、预设目录和轻量引导信息。
  2. 顶部统一展示 3 个 Agent 状态卡，用户可快速切换到 Claude/Codex/Gemini。
  3. 进入 Claude 后，按 `安装检查 -> 选择供应商 -> 充值/API Key 引导 -> 模型/高级项 -> 预览并应用` 向导完成配置。
  4. 用户点击预览后，后端返回 `previewId + normalizedDraft + maskedDiff + warnings + secretRefs`。
  5. 用户确认应用后，后端写入工作区配置、系统凭据库、`~/.claude/settings.json`、Claude saved provider 数据表和 `ai_config_audit_logs`，并发出 `ai_config/changed`。
  6. 用户从 Claude 卡片下的“已保存供应商”子入口进入列表页时，可以查看所有历史应用成功的 Claude 配置，并直接切换当前生效项；切换后工作区配置、live settings 与 active saved provider 必须保持一致。
  7. 用户在 Claude 已保存供应商列表中点击“编辑配置”时，界面必须复用 Claude 详情参数配置页；预览/应用链路需继续走 `preview -> apply -> audit`，并优先复用目标 saved provider 原有的 `savedProviderId` 与托管密钥，不得额外新建重复记录。
  8. 用户启动新的 Claude 会话时，无论是 GT Office 内新终端还是系统级独立 Claude CLI，都会读取同一份 live settings；已运行会话不热切换。
- 异常流程：
  1. workspace 未打开 -> 不允许进入配置流，提示先打开工作区。
  2. Claude 预设缺少 API Key 且无已有 `secretRef` -> `AI_CONFIG_INVALID`。
  3. `previewId` 过期或不存在 -> `AI_CONFIG_PREVIEW_NOT_FOUND`。
  4. 系统凭据库写入失败 -> `AI_CONFIG_SECRET_FAILED`。
  5. `~/.claude/settings.json` 读写失败或结构非法 -> `AI_CONFIG_LIVE_SYNC_FAILED`。
  6. 切换到不存在的 saved provider -> `AI_CONFIG_SAVED_PROVIDER_NOT_FOUND`。
- 验收标准：
  1. Claude 至少 1 个精选预设能完成“预览 -> 应用 -> 更新 `~/.claude/settings.json` -> 启动新会话生效”闭环。
  2. `.gtoffice/config.json` 中不出现 API Key 明文，只出现 `secretRef`。
  3. Codex/Gemini 页面不出现空白复杂表单，且能提供可点击的官方文档与本地配置路径说明。
  4. Claude 向导支持返回上一步、跳过引导、inline 错误反馈和成功提示。
  5. 已安装 CLI 打开向导时默认落在首个可编辑步骤；在 Claude 供应商模式页选择 `official/custom` 后应立即进入对应下一步，不再额外停留在无输入页面。
  6. Claude 卡片必须同时提供“配置向导”和“已保存供应商”两个子入口；saved provider 列表页至少展示 2 条已保存配置，并支持一键切换当前生效供应商；切换时只更新激活标记，不得因为激活状态变化重排列表顺序。
  7. Claude saved provider 列表必须支持“编辑配置”；从列表进入编辑时应复用详情参数配置页，并在应用后继续使用原 `savedProviderId` 更新该条记录，而不是新增一条重复配置。

### US-28 样式系统工程化重构（T-175）

- FR：
  1. 样式入口统一为 `src/styles/index.scss`。
  2. 样式分层：`tokens/foundations/features/utilities`。
  3. 历史变量收敛为设计令牌并保留兼容别名。
- NFR：
  1. 分阶段可发布，迁移过程持续可构建。
  2. 不引入不可控全局副作用。
- 前置条件：`docs/07` 已允许 `sass` 依赖。
- 主流程：
  1. 搭建 SCSS 骨架并切换入口。
  2. 按 feature 迁移并去重。
  3. 执行最小验证链路。
- 异常流程：
  1. 构建失败 -> 回滚到最近可编译状态并定位层级冲突。
  2. 视觉偏差超阈值 -> 补兼容 token，延后视觉重构。
- 验收标准：
  1. 不再新增业务 `.css` 入口。
  2. `npm run typecheck`、`npm run build:web`、`cargo check --workspace` 可通过。

### US-29 角色管理侧栏真实数据与编辑收敛（T-175）

- FR：
  1. 角色管理侧栏列表项仅保留编辑入口，不再直接暴露删除按钮。
  2. 新增角色与编辑角色必须复用同一弹窗。
  3. 删除仅允许在编辑态触发。
  4. 角色管理列表必须基于 SQLite `agents + agent_roles` 真实数据，而非前端写死 seed。
- NFR：
  1. 旧数据库升级不要求手工迁移；缺失字段需自动补 schema。
  2. 角色编辑后无需重启前端即可刷新侧栏与中央画布。
- 前置条件：已绑定 workspace；Tauri runtime 可访问本地 SQLite。
- 主流程：
  1. 前端进入角色管理页时，按当前 `workspaceId` 拉取 `agent.role_list` 与 `agent.list`。
  2. 侧栏列表展示 agent 实例，点击编辑按钮打开复用弹窗。
  3. 新增/编辑提交后分别调用 `agent.create` / `agent.update`，成功后重新拉取列表。
  4. 编辑态点击删除后调用 `agent.delete`，成功后刷新列表并关闭弹窗。
- 异常流程：
  1. 数据库 schema 旧版本缺列 -> `ensure_schema()` 自动补列。
  2. role 不存在 -> 后端返回 `invalid argument: role_id not found`，前端保持现状不落脏数据。
- 验收标准：
  1. 列表项不再显示删除按钮，编辑入口可正常打开弹窗。
  2. 新增、编辑、删除各执行一次后，刷新页面仍能从数据库读到一致结果。
  3. `npm run build` 与 `cargo check --workspace` 通过。
