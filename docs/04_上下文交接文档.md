## 已完成

- 后端已新增 `apps/desktop-tauri/src-tauri/src/connectors/feishu/`：
  - `account_store.rs`：Feishu 账号元数据落 `channel/connectors.json`，并保留 Telegram 字段不被覆盖
  - `api.rs`：通过 Feishu Open API 探测 `tenant_access_token` 与 `bot/v3/info`
  - `inbound.rs`：统一解析 webhook / stream payload 到 `ExternalInboundMessage`
  - `probe.rs`：产出结构化 health，包括 `botName/botOpenId/domain/runtimeConnected/webhookMatched`
  - `webhook.rs`：生成并校验 Webhook callback URL
  - `websocket.rs`：基于 `feishu-sdk` 启动长连接 supervisor，并将 `im.message.receive_v1` 事件转入既有外部通道路由链路
- `commands/tool_adapter/mod.rs` 已补齐 Feishu command 分支：
  - `channel_connector_account_upsert`
  - `channel_connector_account_list`
  - `channel_connector_health`
  - `channel_connector_webhook_sync`
  - `channel_adapter_status`
  - 同时 `resolve_binding_bot_name()` 现在支持 Feishu bot identity 自动回填
- `channel_adapter_runtime.rs` 中的 Feishu 解析已切到 `connectors::feishu::parse_webhook_payload()`，shared runtime 不再自带 Feishu 解析主逻辑
- Feishu external reply 已补齐 finalize 文本出站：
  - `connectors/feishu/api.rs` 新增 `build_client/send_text_message/reply_text_message`
- `connectors/feishu/mod.rs` 新增 `send_text_reply()`
- `commands/tool_adapter/mod.rs` 已把 Feishu 接入 `deliver_external_reply_text()` 和 flush worker
- 设计上禁用了 Feishu preview 流式更新，只在 finalize 阶段发送最终文本
- 本轮已用 Context7 + 本地 `feishu-sdk` 生成表确认并修正 Feishu provider 失败根因：
  - 飞书官方文档对应的 REST 路径是 `POST /open-apis/im/v1/messages` 与 `POST /open-apis/im/v1/messages/:message_id/reply`
  - 但当前 Rust `feishu-sdk` 的 `client.operation(...)` 查的是生成表里的 `operation_id`
  - 该生成表实际只接受单数 `im.v1.message.create` 与 `im.v1.message.reply`
  - 本项目之前误用了复数 `im.v1.messages.create/reply`，因此真实失败为 `unknown operation id`
  - 现已把 `connectors/feishu/api.rs` 改成单数 operation id，并补充定向测试 `sdk_operation_ids_use_singular_message_resource_names`
- Feishu WebSocket supervisor 已修复启动崩溃：
  - 之前 `connectors/feishu/websocket.rs::reconcile()` 在同步上下文里直接 `tokio::spawn`
  - Windows/Tauri 下点击“保存并启动长连接”会报 `there is no reactor running`
  - 现已改为 `tauri::async_runtime::spawn`，并把 worker handle 存储类型切到 `tauri::async_runtime::JoinHandle`
- 前端 `tool-adapter` 已完成模块化改造：
  - `ChannelWizard.tsx` 改为 channel dispatcher
  - 新增 `apps/desktop-web/src/features/tool-adapter/feishu/`
  - Feishu 向导已拆成 `FeishuConnectorWizard/FeishuPlatformGuide/FeishuAccountForm/FeishuHealthCard/model`
  - 当前 UI 已进一步收敛为独立的 Feishu 大尺寸弹窗，且仅保留 WebSocket 接入路径
  - 配置流已按飞书长连接真实要求调整为“开放平台创建应用拿凭据 -> 在 GT Office 启动长连接并确认 runtime connected -> 回开放平台保存长连接订阅 -> 回 GT Office 完成 route / policy”
  - 开放平台入口改成“复制 URL 手动打开浏览器”，不再依赖自动拉起浏览器
  - `FeishuConnectorWizard` 的连接测试现在会轮询 health，只有 `health.ok && runtimeConnected` 才允许进入下一步，避免在飞书后台保存长连接时报“未检测到应用连接信息”
  - Channel 配置入口已从 `SettingsModal` 脱离，改为顶栏进入独立 `Channel Studio`，避免机器人接入流程继续嵌在系统设置弹窗里
- Feishu health 提示已增强：
  - 当开放平台返回 `bot capability is not activated` 时，UI 会直接展示“开启 Bot 能力 + 发布最新版本 + 回 GT Office 重试”的检查清单
- Feishu 入站账号映射已修复：
  - 之前 WebSocket 事件会把飞书 `app_id/tenant_key` 写进 `ExternalInboundMessage.account_id`
  - route binding 与 access policy 实际按 GT Office connector `accountId`（通常为 `default`）匹配
  - 这会导致消息成功入站但统一落到 `CHANNEL_ROUTE_NOT_FOUND`
  - 现已新增 `connectors/feishu::parse_payload_for_account()`，并在 websocket handler 中显式注入当前 connector account id，确保 Feishu 长连接事件与已保存 binding/policy 使用同一账号维度
- Feishu 外部回复 relay 绑定已修复：
  - 之前 `commands/tool_adapter/mod.rs::bind_external_reply_sessions()` 内部硬编码 `channel != "telegram" => return`
  - 结果是 Feishu 消息虽然已经成功入站、路由、派发到 agent，但不会建立 external reply session
  - flush worker 的 Feishu finalize 出站代码其实已经存在，只是永远拿不到候选 session
  - 现已抽取 `channel_supports_external_reply()` 并将 Feishu 一并纳入，确保 Feishu 与 Telegram 都会绑定 reply relay session
- 参考 OpenClaw 已继续收口 Feishu 出站实现：
  - OpenClaw 的关键点不是“永远直接发 chat”，而是先尝试 `message.reply`，但 `msg_type` 放在 request body 中；reply 目标异常时再 fallback 到 `message.create`
  - 本项目此前 `reply_text_message()` 把 `msg_type=text` 放在 query 参数，且没有 reply fallback；现在已改为 body 内 `msg_type`
  - `connectors/feishu/mod.rs::send_text_reply()` 现在会在 reply 失败时自动降级为对当前 `peer_id(chat_id)` 直接发送，尽量保证用户至少能看到最终回复
- 前端外部事件观测已补齐：
  - 之前“最近外部事件”只展示 `external/channel_reply`，这个实际上对应内部 `channel/ack`
  - Feishu 是否真的把消息发出去，用户在 UI 上是看不到 `external/channel_outbound_result` 的
  - 现在 `ShellRoot` 已追加 `onExternalOutboundResult` 订阅，事件面板新增 “出站 / Outbound” 类型，后续可直接区分“内部派发成功”和“飞书真实出站成功/失败”
- 外部事件面板的早到 `delivered` 误导已修复：
  - 后端 `emit_channel_events()` 会把内部 `channel/ack` 同时镜像成 `external/channel_outbound_result`，其 `relayMode=dispatch-ack`
  - 之前前端没有消费 `relayMode`，因此会把“任务已写入 agent terminal”误显示成“飞书已出站 delivered”
  - 现在 `desktop-api` 已补齐 `relayMode/confidence` 字段，`ShellRoot` 会忽略 `relayMode=dispatch-ack` 的 outbound 事件
  - 结论：后续“最近外部事件”里的 `出站` 只代表真实外部发送结果；内部派发 ack 仍保留在 `回执`
- `回执 / Reply` 事件本身也已确认不代表 agent 回复：
  - `external/channel_reply` 来自 `emit_channel_events()`，本质仍是内部 `channel/ack` 的镜像
  - 这类事件会与 `sending/sent` 同秒出现，因为它发生在任务写入 agent terminal 之后，而不是 agent 产生最终输出之后
  - 现在前端已不再把 `external/channel_reply` 展示在“最近外部事件”里，避免继续误读为“飞书机器人已经回复用户”
- 本轮又确认了一条更深的 Feishu 丢回复路径，并已修复：
  - `AppState::external_reply_session_text()` 之前只要 session 收到过任意 rendered-screen，就会无条件返回 `last_rendered_reply_text`
  - 如果该次屏幕提取因为 prompt/status bar/noise 过滤结果为空，后端不会再使用 VT buffer 文本
  - 这会让 reply session 在 `take_external_reply_dispatch_candidates()` 里被判定为“到达 finalize 边界但没有正文”，最终静默 drop，不会发出真实 `external/channel_outbound_result`
  - 现已改成“rendered text 非空优先；为空时回退 VT parser + normalize_reply_text()”，同时 finalize 判定对“rendered 为空但 PTY 有正文”的 session 不再强依赖 ready prompt；新增定向回归测试 `bound_reply_session_finalizes_with_pty_fallback_when_rendered_extract_is_empty`
- 外部事件面板已按联调需求进一步收口：
  - “最近外部事件”现在只展示 `入站 / 出站`
  - `入站` 直接展示飞书消息正文摘要，辅助信息保留 `channel/sender/peer/msg`
  - `出站` 直接展示真实发往外部通道的正文摘要；失败时才显示错误 detail
  - `路由 / 派发 / reply / error` 中间态已从该面板移除，避免继续干扰对真实收发闭环的判断
  - 对同一条失败 `出站` 已新增 `mergeKey` 合并，外部事件面板不再重复刷同一失败，只更新最新时间和错误 detail
- `desktop-api.ts` 已扩展 Feishu connector DTO 字段；`ChannelManagerPane` 健康检查兼容 `botName ?? botUsername`
