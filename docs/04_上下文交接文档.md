# vbCode 上下文交接文档

## 当前里程碑与本轮目标

- 里程碑：M2 MVP 完整
- 任务：T-171 多 Agent MCP 桥接
- 本轮目标：把 MCP 公共协作模型切到“两步式”目录/发送，保证当前 agent 身份能自动同步进 MCP，并保持 `dispatch/handover` 真流可用

## 已完成

- 新增 public MCP tool `gto_get_agent_directory`
  - 先拉取当前 workspace 的组织目录，再按 `agent_id` 做协作
  - 返回 `departments/roles/agents/runtimes/self`
  - `agents[]` 现已合并 repo agent 与 runtime-only agent，headless/bootstrap 场景也能看到在线 agent
- 在 `apps/desktop-tauri/src-tauri/src/mcp_bridge.rs` 新增 `directory.get`
  - bridge 内存缓存目录 snapshot
  - 同步写入 `.gtoffice/mcp/directory.json`
  - `gto_health` 现在可看到 `directorySnapshotCount`
- `agent_create`、`agent_runtime_register`、`agent_runtime_unregister` 现在会主动刷新目录 snapshot
- agent/station terminal 创建时注入：
  - `GTO_WORKSPACE_ID`
  - `GTO_AGENT_ID`
  - `GTO_ROLE_KEY`
  - `GTO_STATION_ID`
- `gto_report_status` / `gto_handover` 现在支持 sender 自动推断
  - 显式 `sender_agent_id` 优先
  - 否则读取当前 terminal env 的 `GTO_AGENT_ID`
- `tools/gto-agent-mcp` Node 版与 `gto-agent-mcp-sidecar` Rust 版都已同步：
  - 新增 `gto_get_agent_directory`
  - `gto_health` 返回 `self + directory` 摘要
  - `directory.json` fallback 支持 WSL/Windows 候选路径
  - `workspace_id` 现在可省略，优先从 env，否则自动选最近 directory snapshot
  - public tool description 已显式引导“先查目录、复用 workspaceId、发送前先看 bridgeAvailable”
  - `gto_dispatch_task` 现已明确为“写入目标 terminal 并自动提交执行”；`gto_report_status/gto_handover` 明确为协作消息记录，不负责向目标 terminal 注入命令
  - 发送类 tool 在 bridge 掉线时改为返回 `MCP_BRIDGE_SEND_UNAVAILABLE`，明确说明“目录可读但发送不可用”
  - runtime 与 directory 现在会优先从“包含目标 workspace snapshot 的同一状态根”解析，避免 WSL 读到 Windows 目录、却仍拿 Linux 死 runtime 发消息
  - public tool description 与返回值新增环境说明，明确“Windows 桌面端 + WSL agent/MCP”是支持场景
- `task.dispatch_batch` 与 MCP bridge 终端投递逻辑现在已对齐外部通道链路
  - 先写命令
  - 再发送 runtime `submitSequence`
  - 再补一次硬回车 `\r` 兜底
  - 目标：避免某些交互式 CLI 只显示文本、不真正执行
- 更新 `gto-agent-mcp-probe` 与 `mcp-e2e`
  - probe 先校验 `gto_get_agent_directory`
  - `gto_handover` 不再显式传 sender，改为验证 env 自动识别
- 已真实跑通完整 E2E
  - `gto_get_agent_directory` 成功返回 `agentCount=2`
  - `gto_dispatch_task` 返回 `sent=1 failed=0`
  - `gto_handover` 返回 `senderResolvedFrom="env"` 与 `acceptedTargets=["probe-manager"]`
- 使用 `mcp-builder` 复盘 MCP 桥接设计，确认原始方向正确，核心缺口在 E2E 引导链与发布形态，不在 tool surface 本身
- 修复 `scripts/dev/mcp-e2e.sh`
  - 启动前记录 runtime mtime
  - 仅在 runtime 文件被本次运行刷新后才继续
  - 新增 bootstrap health 轮询，要求 `gto_health` 真正成功
  - 默认走完整 bootstrap + `dispatch/handover` 真流，而不是 health-only
- 改造 MCP 安装入口
  - `tools/gto-agent-mcp/src/installer.mjs` 新增 `local/npx/auto`
  - `tools/gto-agent-mcp/bin/gto-agent-mcp-install.mjs` 新增 `--mode`
  - `apps/desktop-tauri/src-tauri/src/mcp_bridge.rs` 支持 debug/dev 默认 local、release 默认 `npx -y @gtoffice/agent-mcp-bridge@0.1.0 serve`
- 在 `apps/desktop-tauri/src-tauri/src/mcp_bridge.rs` 增加内部 `dev.bootstrap_agents`
  - 打开 workspace
  - 创建 terminal session
  - 注册在线 runtime
  - 返回 workspace/session/agent bootstrap 结果给 probe
- 在 `tools/gto-agent-mcp/bin/gto-agent-mcp-probe.mjs` 增加 bootstrap 流程
  - 先通过 bridge 建立 manager/worker runtime
  - 再调用真实 MCP tool `gto_dispatch_task` 与 `gto_handover`
- 修复 `crates/vb-terminal/src/lib.rs`
  - 去掉 runtime 线程里的 `blocking_send`
  - 改为 runtime-safe 的 `try_send`
  - 解决 `dev.bootstrap_agents` 触发的 panic：`Cannot block the current thread from within a runtime`
- 补齐 `crates/vb-task/tests/lib_tests.rs` 在线目标 `handover` 成功回归
- 已真实跑通完整 E2E
  - bootstrap 成功创建 `probe-manager` / `probe-worker-1`
  - `gto_dispatch_task` 返回 `sent=1 failed=0`
  - `gto_handover` 返回 `acceptedTargets=["probe-manager"]`

## 风险 / 阻塞

- 当前仓库仍是脏工作树，且有大量与本任务无关的改动；后续不要误回滚
- `dev.bootstrap_agents` 目前是内部 bridge 方法，适合本地 probe/E2E；若未来外部依赖它，需要先明确边界与安全策略
- 发布态 `npx` 安装路径已具备，但 npm registry 上的包版本、安装失败回滚和版本锁定策略还没最终产品化
- probe/headless 目录里 runtime-only agent 的 `roleKey` 目前可能为空；正式站点 terminal 已注入 role key，但 bootstrap 如要更贴近真实组织，还需要补一层 probe role 映射
- 当前真实 GT Office bridge 运行在 Windows 侧 `127.0.0.1:50568`；WSL 可直连该端口，真正的失败原因已确认是旧版 sidecar 误选 Linux 侧 `127.0.0.1:45423`
