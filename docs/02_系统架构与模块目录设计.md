# GT Office 系统架构与模块目录设计

## 1. 架构原则

1. Rust 负责重任务与系统能力，WebUI 负责渲染与交互。
2. 模块边界优先于功能堆叠，避免跨层耦合。
3. 所有耗时任务统一进入任务系统（可取消、可观测、可恢复）。
4. 跨平台差异收敛在基础设施层，不泄漏到业务层。
5. 多 Agent 协作能力必须可治理（可授权、可审计、可回放）。
6. 先保证本地单机可靠，再按阶段引入 Redis/分布式能力。

## 2. 分层架构

- Presentation Layer（WebUI）
  - 页面、组件、状态管理、快捷键、主题、设置中心。
- Application Layer（Tauri Commands + Use Cases）
  - 前后端命令入口、会话协调、权限校验、配置变更编排。
  - 对于重任务场景（大规模搜索、海量终端、百万文件目录），Tauri 仅做 sidecar 编排，不执行重逻辑。
- Sidecar Layer（Rust Core Daemon）
  - 通过 UDS/Named Pipe/localhost socket 提供高性能二进制协议服务。
  - 承载 workspace/fileio/search/git/terminal 等重逻辑与流式事件。
- Domain Layer（Core Services）
  - Workspace、WorkspaceContext、File、Terminal、Git、Tool Adapter、Change Feed、Settings、Keymap、AI Config Policy。
  - Agent Organization、Agent Manager、Message Channel、Hook Engine、Policy Engine、Agent Observability。
- Infrastructure Layer
  - PTY、文件监听、数据库、系统凭据、系统 git、日志、系统配置目录适配。
  - Redis Adapter（可选阶段）、Event Bus、Schema Registry、Telemetry Pipeline。

## 4. 关键模块职责

### 4.1 `vb-workspace`

- 管理工作区元数据、最近项目、会话恢复。
- 读写 `.gtoffice/config.json`。
- 管理 `WorkspaceContext`（workspace_id/root/permissions/active state）。

### 4.2 `vb-filesystem`

- 文件树读取、缓存、懒加载。
- 文件监听（notify）与增量刷新。
- 搜索能力（rg 适配、后续可接 Tantivy）。

### 4.3 `vb-terminal`

- PTY 创建、输入输出流、会话生命周期。
- shell/cwd/env 注入、会话快照。
- 会话必须绑定 `workspace_id`，默认 cwd 为工作区根目录。

### 4.4 `vb-git`

- status/diff/log/branch 读取。
- 提交、暂存、stash 与复杂命令 fallback。

### 4.5 `vb-tools`

- Tool Adapter profile 管理。
- 启动命令拼装、参数模板、权限策略。

### 4.6 `vb-changefeed`

- 文件变更事件归因（来源会话、时间、摘要）。
- 与 Git/Terminal 事件联动。

### 4.7 `vb-task`

- 统一任务队列与执行状态。
- cancel token / timeout / retry / 日志关联。

### 4.8 `vb-security`

- 工作区授权与路径白名单。
- 密钥安全存储与读取。
- 危险操作策略控制。

### 4.9 `vb-settings`

- 统一配置源管理（默认/用户/工作区/会话）。
- 配置 schema 校验与版本迁移。
- 配置来源追踪与变更通知广播。

### 4.10 `vb-keymap`

- 快捷键绑定与平台映射（Cmd/Ctrl 差异）。
- 冲突检测、优先级计算与回退策略。
- 快捷键变更热更新。

### 4.11 `vb-ai-config`

- AI 配置访问白名单策略（可读/可写字段隔离）。
- 配置改动预览、确认、应用流水线。
- 审计日志与一键回滚。

### 4.12 `vb-abstractions`

- 定义跨模块抽象接口（trait），隔离实现差异。
- 提供核心契约：
  1. `WorkspaceScopedService`
  2. `TerminalProvider`
  3. `GitProvider`
  4. `SettingsStore`
  5. `CommandPolicyEvaluator`
- 支持 mock 实现用于单元测试与回归验证。

### 4.13 `vb-agent`

- 管理 Agent 生命周期（create/start/pause/resume/terminate）。
- 绑定 Agent 与岗位角色（`agent_id`/`role_id`/`workspace_id`）。
- 调度任务分配与执行归属。

### 4.14 `vb-channel`

- 提供结构化通信通道与消息协议。
- 提供同通道有序、去重、重试与确认机制。
- 支持窗口间/工作区间消息桥接。

### 4.15 `vb-hook`

- 提供事件订阅注册、过滤与动作执行。
- 支持并发、超时、重试、熔断与幂等策略。
- 与 `vb-task` 集成，统一任务可观测性。

### 4.16 `vb-policy`

- 统一岗位职责、权限策略与上下文约束评估。
- 输入来源包括 Markdown Charter + JSON Policy 快照。
- 对命令执行、文件访问、配置写入进行策略裁决。

### 4.17 `vb-observability`

- 聚合 Agent-Task-Message-ChangeFeed 关联图数据。
- 输出时间线与图视图查询模型，供前端可视化。
- 负责异常链路识别（失败热点、重试风暴、越权操作）。

### 4.18 `vb-cache`（可选阶段）

- 封装 Redis 访问（缓存、Pub/Sub、临时状态）。
- 提供本地降级模式（Redis 不可用时回退 SQLite/内存队列）。
- 不承载权威数据写入，仅做加速与实时分发。
- 当前决策：M1/M2 不落地 `vb-cache` 运行时实现，仅保留接口与文档契约。

## 5. 前后端接口边界

- WebUI 只通过 Tauri Command/Event 与后端交互。
- `packages/shared-types` 维护 request/response/event 类型。
- 事件建议：
  - `terminal/output`
  - `task/changed`
  - `changefeed/append`
  - `workspace/updated`
  - `workspace/active_changed`
  - `settings/updated`
  - `keymap/updated`
  - `ai_config/changed`
  - `agent/state_changed`
  - `channel/message`
  - `hook/executed`
  - `policy/denied`
  - `observability/graph_updated`

## 7. 模块间依赖约束

1. `vb-core` 可依赖各领域模块；领域模块不得反向依赖 UI。
2. `vb-terminal` 不依赖 `vb-git`；通过 `vb-task` 和事件总线协作。
3. `vb-security` 作为横切模块，不承载业务流程编排。
4. `vb-ai-config` 只能通过 `vb-settings` 写配置，不直接写文件。
5. 所有 workspace 范围内操作必须显式传递 `workspace_id`。
6. 终端启动请求若未携带 `workspace_id` 必须拒绝。
7. UI Feature 之间只通过 store/action 或 command 调用协同。
8. `vb-policy` 是命令执行前的强制前置校验，不得绕过。
9. Redis 仅可通过 `vb-cache` 访问，业务模块不得直接使用 redis client。
10. Agent 相关写操作必须包含 `agent_id` 与 `role_id`。

## 8. 工程化抽象约束（封装/继承/多态）

1. 依赖倒置：上层依赖 `vb-abstractions`，不依赖具体基础设施实现。
2. 封装边界：文件/Git/终端的底层细节仅在各自模块内部可见。
3. 继承与扩展策略：
   - Rust 后端以 trait + 默认实现（模板方法）表达“抽象基类”能力。
   - 前端组件允许基类/组合式 Hook 复用，但业务逻辑优先组合而非深继承。
4. 多态扩展：Provider 可替换（例如 `git2` 与 system git fallback）而不影响上层用例。
5. 可测试性：每个核心服务需有 mock provider 的单元测试入口。

## 9.1 Agent 协作数据流（新增）

1. `AgentManager` 接收任务分配 -> 创建 `task_run`。
2. `Channel` 发布任务消息 -> Agent 消费并执行。
3. 执行产生 `terminal/git/fs` 事件 -> 进入 `HookEngine`。
4. `HookEngine` 触发自动化动作并写入 `hook_runs`。
5. `PolicyEngine` 对关键动作做授权裁决并记录 `policy_decisions`。
6. `Observability` 聚合为关系图并推送前端视图。

## 9. 测试策略

1. Rust 单元测试：领域模块独立验证。
2. 集成测试：命令层到服务层链路。
3. E2E：关键路径（打开工作区、终端启动、Git 提交、恢复会话、多工作区切换）。
4. 性能回归：大仓库文件树与搜索基准测试。
5. Agent 协作回归：岗位权限、消息顺序、Hook 熔断、交接策略校验。
