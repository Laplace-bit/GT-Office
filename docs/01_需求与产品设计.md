# GT Office 需求与产品设计

## 1. 项目定位

- 产品定位：面向 AI Coding 场景的跨平台桌面开发工具。
- 产品升级方向：从“单人 Vibe Coding 工具”升级为“多 Agent 协作操作系统（Agent Office OS）”，让多个 Agent 像团队成员一样分工、通信、协作与审计。
- 核心目标：在“文件管理 + Terminal + Git + 多窗口并行”能力上，针对 Codex / Claude Code / aider 等 CLI 工作流做到深度优化。
- 平台范围：Windows / Linux / macOS。
- 体验目标：具备现代 IDE 体验，同时保持更轻量、更快启动、更低资源占用。
- 视觉目标：遵从“苹果”风格，简洁、高级、专业，减少装饰性噪音并强化信息秩序。

## 2. 技术路线

- 宿主框架：Tauri。
- 核心能力层：Rust（文件系统、PTY、Git、索引、任务调度、安全控制）。
- UI 层：WebUI（建议 React + Tailwind + shadcn/ui，或 SvelteKit）。
- 数据存储：SQLite + 项目级 `.gtoffice/config.json`。

## 3. 用户与核心场景

- 个人开发者：并行运行多个 AI CLI 会话，快速完成修复/重构/测试。
- 团队开发者：共享项目配置与工作流模板，降低环境差异。
- 重度终端用户：在一个工具中完成文件、命令、Git 与变更追踪。

关键场景：

1. 在同一项目中同时运行多个 AI CLI 会话并可追踪改动来源。
2. 在大仓库中快速搜索/定位/打开文件，并与 Git 状态联动。
3. 切换窗口继续任务，能恢复上下文并无缝接力。
4. 通过设置中心和快捷键快速个性化工作流，并允许 AI 在可控边界内直接调整配置。
5. 同时打开多个工作区，且每个终端会话默认在对应工作区根目录启动，避免误操作到错误目录。
6. 多 Agent 并行执行不同岗位职责（实现、评审、测试、发布），通过统一通信通道与任务系统协同推进。
7. 通过 Hook 订阅机制在关键事件（文件变更、提交、任务状态）触发自动化流程。
8. 在可视化面板中查看 Agent 信息流、任务流、决策链路，确保“并行但可控”。

## 4. 范围边界（Scope）

### 4.1 In Scope（首期必须）

1. Workspace 管理（多工作区、最近项目、会话恢复）。
2. 文件树与文件操作（含搜索、重命名、拖拽、监听刷新）。
3. 真 PTY Terminal（xterm.js + portable-pty，多标签/分屏/会话）。
4. Git 基础能力（status/diff/commit/branch/stash）。
5. Tool Adapter（至少支持 Codex 与 Claude Code 一键启动）。
6. 多窗口并行 + 全局任务中心。
7. Change Feed（文件改动归因到会话）。
8. 安全模型（目录授权、命令边界、密钥安全存储）。
9. 设置中心（用户级/工作区级配置，支持导入导出）。
10. 快捷键系统（可视化编辑、冲突检测、按平台适配）。
11. AI 开放配置能力（AI 可读写允许范围内的配置，并保留审计记录）。
12. 多工作区上下文隔离与终端目录绑定（Terminal 必须绑定 workspace 上下文）。

### 4.2 Out of Scope（后续迭代）

1. 完整云协作与实时多人编辑。
2. 完整插件市场与远程分发体系。
3. 企业级 SSO/组织域集成与跨组织治理平台。

## 5. 功能需求与验收标准

### FR-01 Workspace 管理

- 支持创建/打开/关闭工作区。
- 支持同时打开多个工作区（单窗口多工作区模式 + 多窗口模式）。
- 支持最近项目与固定项目。
- 支持重启恢复：窗口布局、终端会话、上次打开文件。
- 支持激活工作区切换，所有上下文面板（文件/Git/终端）按激活工作区联动。

验收标准：

1. 重启后 90% 以上 UI 布局状态可恢复。
2. 工作区配置可写入 `.gtoffice/config.json`。
3. 多工作区并行打开时，状态、任务、终端会话互不串扰。

### FR-02 文件管理

- 左侧文件树：支持 Git 状态徽标、忽略规则、懒加载。
- 文件操作：新建、删除、重命名、移动、批量操作、拖拽。
- 快速打开（Ctrl/Cmd+P）与全局文本搜索（rg）。
- 文件监听：外部变更实时刷新。

验收标准：

1. 10 万文件规模仓库内，文件树首屏渲染可用。
2. 搜索结果支持增量展示与中断取消。

### FR-03 Terminal（真终端）

- 支持 ANSI、光标控制、交互式 TUI。
- 支持多标签、多分屏、每窗口多会话。
- 支持会话配置：shell、cwd、env 注入。
- 新建终端默认在目标工作区根目录启动（`cwd = workspace.root`）；只有显式授权时才允许自定义 cwd。
- 提供工具模板：codex/claude/aider/gitui。

验收标准：

1. `fzf` / `gitui` / `vim` 等交互程序可正常使用。
2. 粘贴大段文本时有安全提示或分块处理。
3. 任意终端会话都可追溯所属 workspace_id 与 resolved cwd。

### FR-04 Git 集成

- 基础读取：status/diff/log/branch（`git2`）。
- 复杂流程：回落系统 git（rebase/merge 特殊场景）。
- 支持 staged/unstaged、提交、stash、branch 切换。

验收标准：

1. Git 面板状态与 `git status` 一致。
2. 失败操作有可读报错与回滚提示。

### FR-05 Tool Adapter

- 每个工具定义：`cmd`、`args` 模板、`env`、安全策略。
- 一键启动 AI CLI 会话并绑定工作区。
- 可将“选中文件/目录/最近 diff”注入上下文。

验收标准：

1. 新增一个工具适配不需要改核心流程代码。
2. 工具配置可导入导出。

### FR-06 多窗口与并行任务

- 支持多窗口独立运行工作区。
- 全局任务中心展示所有窗口任务状态。
- 支持窗口间切换与任务跳转。

验收标准：

1. 多窗口互不阻塞，关闭单窗口不影响其他任务。
2. 任意窗口可查看全局任务概览。

### FR-07 Change Feed

- 记录文件变更事件（时间、来源会话、变更摘要）。
- 提供快速 diff、打开文件、回滚入口。

验收标准：

1. 变更事件可回溯到具体终端会话或工具实例。
2. 时间线可按工作区/会话筛选。

### FR-08 安全与权限

- 首次打开工作区要求目录授权。
- 默认命令执行边界限制在 workspace 内。
- 密钥使用系统凭据存储（Keychain/Credential Manager/libsecret）。
- 高危操作需二次确认。

验收标准：

1. 未授权目录不可被命令执行访问。
2. 密钥不落盘明文。

### FR-09 设置中心与本地配置

- 提供设置页面，支持搜索、分组、重置、导入导出。
- 支持配置分层：默认配置 < 用户全局配置 < 工作区配置 < 临时会话覆盖。
- 用户全局配置存储于系统标准目录；工作区配置存储于 `.gtoffice/config.json`。
- 支持常用 IDE 精简项：主题、字体、终端默认 shell、启动行为、搜索忽略规则、工具 profile 默认值。

验收标准：

1. 设置修改后可在 1 秒内生效（无需重启的配置）。
2. 能清晰展示配置来源（默认/用户/工作区/会话）。
3. 导出配置可被同版本客户端导入并成功校验。

### FR-10 快捷键系统

- 提供快捷键页面，支持按命令搜索、录制按键、平台差异映射（Win/Linux/macOS）。
- 支持冲突检测与一键修复建议。
- 提供 VSCode 兼容的常用默认键位集，并允许恢复默认。
- 支持“工作流级快捷操作”：启动指定 Tool Profile、聚焦终端、打开 Change Feed、切换任务中心。

验收标准：

1. 冲突快捷键在保存前必须阻断并提示影响命令。
2. 核心操作可通过键盘完成（打开文件、切终端、提交、启动工具）。
3. 快捷键变更即时生效并可回滚。

### FR-11 AI 可读写配置（Vibe Config）

- 为 AI 提供受控配置访问接口，不直接暴露全部本地敏感配置。
- AI 可读取：主题、布局、快捷键、工具 profile 非敏感字段、工作区偏好。
- AI 可写入：允许清单中的配置项，写入前先做 schema 校验与策略校验。
- 所有 AI 配置改动必须记录审计日志（时间、来源会话、改动前后 diff、操作者）。
- 提供“预览改动 -> 用户确认 -> 应用”流程；支持自动批准白名单项。

验收标准：

1. AI 无法读取或修改未授权配置项（如密钥、系统凭据）。
2. 每次 AI 改动都能在审计面板回溯并可一键回滚。
3. 配置写入失败返回稳定错误码并不污染现有配置文件。

### FR-12 设计系统与视觉规范

- 建立统一设计令牌（色彩、间距、圆角、阴影、动效、字体层级）。
- 视觉风格遵循：克制、留白、强对齐、高信息密度可读性。
- 动效采用快速、低干扰过渡，避免花哨动画。
- 支持浅色/深色主题，但保持统一品牌语言与专业质感。

验收标准：

1. 主界面视觉一致性通过设计检查清单（色彩/间距/层级/反馈）。
2. 关键路径页面（工作区、终端、设置）符合统一组件规范。
3. 页面交互响应与动效不影响高频操作效率。

### FR-13 工程化抽象与可扩展架构

- 核心能力必须通过稳定抽象层暴露，避免 UI 与基础设施直接耦合。
- 采用“接口/抽象基类（或 trait）+ 组合优先 + 多态扩展”实现可替换能力。
- 工作区相关能力必须实现统一上下文接口（WorkspaceContext），禁止无上下文访问文件/Git/终端。
- 对外部依赖（PTY、Git、配置存储）提供可替换 Provider，支持 mock 与 fallback。

验收标准：

1. 关键服务（workspace/terminal/git/settings）具备接口层与实现层分离。
2. 多工作区相关操作必须显式传递 `workspace_id`，缺失即失败。
3. 单元测试可通过 mock provider 验证核心流程。

### FR-14 Agent 组织与岗位治理

- 支持定义 Agent 组织结构：团队、岗位、职责模板、能力边界。
- 支持岗位级权限模型：可执行命令范围、可访问目录、可读写配置域、是否允许网络。
- 支持 Agent Manager：创建/暂停/恢复/终止 Agent 会话，分配任务并追踪执行状态。

验收标准：

1. 任意 Agent 会话都必须绑定 `agent_id`、`role_id`、`workspace_id`。
2. 越权操作必须被策略引擎拦截并返回稳定错误码。
3. 任务中心可按 Agent/岗位维度筛选执行记录。

### FR-15 Agent 通信通道与协作协议

- 提供结构化通信通道（Channel）：Direct、Group、Broadcast。
- 消息类型支持：任务指令、状态更新、变更通知、阻塞求助、交接消息。
- 支持消息可靠投递、重试、去重与顺序保证（至少同通道内有序）。

验收标准：

1. 同一通道消息序列号连续，丢失可检测。
2. 任务交接消息可关联到具体任务 ID 与提交记录。
3. 关闭单个窗口不会导致全局通道消息丢失。

### FR-16 Hook 订阅与自动化编排

- 提供 Hook 总线，支持事件订阅：`workspace.*`、`terminal.*`、`git.*`、`task.*`、`changefeed.*`、`agent.*`。
- 支持 Hook 执行动作：触发命令、生成任务、发送通知、调用外部 Webhook（受安全策略约束）。
- 支持 Hook 策略：并发限制、超时、重试、熔断、幂等键。

验收标准：

1. Hook 失败不阻塞主流程，必须有隔离与告警。
2. 单事件可配置多个订阅者并独立执行。
3. Hook 执行链路可追溯（事件 -> 订阅 -> 动作 -> 结果）。

### FR-17 高速数据面与 Redis 支撑（未来核心）

- 引入 Redis 作为大规模高频状态缓存与实时通道支撑（阶段化启用）。
- 使用场景：任务队列缓存、Agent 在线状态、事件流暂存、可视化实时指标聚合。
- 持久化权威数据仍由 SQLite/本地存储承担，Redis 作为加速层而非唯一事实来源。
- 当前决策：M1/M2 阶段不实现 Redis，仅保留接口与架构扩展位，默认走本地后端。

验收标准：

1. Redis 不可用时系统可降级回本地队列模式。
2. 缓存键具备 TTL 与命名规范，避免无限增长。
3. 关键状态在 Redis 与权威存储间具备一致性修复机制。

### FR-18 Agent 信息流可视化

- 提供 Agent Graph 视图：展示 Agent、任务、消息、Hook、文件变更之间的关系。
- 支持时间轴回放与过滤（按工作区、Agent、任务、事件类型）。
- 支持异常高亮：失败链路、重试风暴、权限拒绝热点。

验收标准：

1. 可从任一变更事件追溯到触发 Agent 与上游消息。
2. 图视图在中等规模（1 万事件）下保持可交互。
3. 可视化结果与审计日志字段一一对应。

### FR-19 Markdown 驱动的岗位职责与策略（Agent Charter）

- 支持以 Markdown 文档定义 Agent 岗位职责、操作边界、交接要求与 DoD。
- 启动 Agent 时自动加载并解析对应 Charter，转为可执行策略快照。
- 支持版本化与差异审查，确保策略变更可追溯。

验收标准：

1. 无 Charter 或 Charter 校验失败时，Agent 不可进入执行态。
2. Charter 变更可触发策略热更新或按会话生效策略。
3. 交接规则（更新 `docs/03` + `docs/04`）可作为策略项被自动校验。

### FR-20 Agent First 主布局（工位优先）

- 主界面布局以 Agent 工位与终端执行为核心，不采用传统编辑器中心模型。
- 每个 Agent 对应一个岗位实体，每个命令行窗口对应该 Agent 的工位（终端会话主入口）。
- 中央区域必须支持工位矩阵视图与工位详情视图切换，并优先保证中央画布宽度与终端可视空间。
- 文件管理、Git、变更流、AI 上下文作为协同能力围绕工位运行，而非依赖独立右侧面板。
- 百级 Agent 规模下需支持虚拟化渲染、分级日志订阅、工作区级原子切换。

验收标准：

1. UI 骨架中可明确看到“工位矩阵 + 左侧业务面板 + 状态栏”核心区域，中央画布占比优先。
2. 任意工位都可追溯 `agent_id/role_id/workspace_id/terminal_session_id`。
3. 多工作区切换后，新建终端始终绑定激活工作区根目录，且工位状态不串扰。
4. 百级工位列表滚动与筛选保持可交互，不出现明显卡顿。

## 6. 非功能需求（NFR）

1. 性能
   - 冷启动目标：< 2.5s（中等机器）。
   - 常驻内存目标：显著低于同类 Electron IDE。
   - 大仓库文件树操作保持可交互，不出现长时间主线程阻塞。
2. 可靠性
   - 任务具备取消、超时、重试策略。
   - 异常统一日志（`tracing`）并可导出诊断包。
3. 可维护性
   - 模块边界清晰，前后端协议可版本化。
4. 可扩展性
   - Tool Adapter 插拔式注册。
   - Git / 搜索 / 索引能力可分阶段升级。
   - Agent/Hook/消息通道采用插件化扩展点，支持后续能力扩展。
5. 易用性与一致性
   - 设置项命名统一、可搜索、可解释。
   - 键盘优先场景的端到端可达性满足高频开发需求。
6. AI 可治理性
   - AI 配置访问遵循最小权限原则并可审计、可回滚。
7. 工程化质量
   - 核心模块遵循高内聚低耦合，可替换实现不影响上层业务。
   - 多工作区上下文隔离与目录边界约束默认开启。

## 7. 里程碑规划

### MVP（4-8 周）

1. Workspace + 文件树 + 基础文件操作。
2. 真 PTY 终端（多标签）。
3. Git status/diff/commit。
4. 多窗口基础能力 + 多工作区并行管理。
5. Tool Adapter（Codex/Claude 至少 2 个）。
6. 设置中心 MVP（基础设置项 + 配置分层生效）。
7. 快捷键 MVP（查看、修改、冲突检测）。

### V1

1. Change Feed。
2. 会话恢复与布局持久化。
3. 安全模型与凭据存储完善。
4. Git 分支/stash/日志增强。
5. AI 可读写配置通道（含审计与回滚）。
6. 设计系统固化与高一致性 UI 升级。

### V2

1. 语义索引与检索。
2. 插件系统开放。
3. 团队级 profile 共享。
4. Agent 组织治理与岗位职责（Charter）正式上线。
5. Hook 订阅编排与自动化策略引擎上线。

### V3

1. Redis 数据面与实时消息总线增强。
2. Agent 信息流可视化（Graph + Timeline）与异常洞察。
3. 组织级协作能力（团队模版、审计报表、跨项目治理）。

## 8. 风险与对策

1. PTY 跨平台差异
   - 对策：封装统一 session 抽象，建立平台兼容测试矩阵。
2. Git 行为差异
   - 对策：`git2` + 系统 git 双通道，失败自动降级。
3. 大仓库性能波动
   - 对策：虚拟列表、懒加载、缓存、任务限流。
4. 并行会话可控性不足
   - 对策：Change Feed + 全局任务中心 + 风险命令确认。
