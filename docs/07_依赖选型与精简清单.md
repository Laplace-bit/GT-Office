# GT Office 依赖选型与精简清单

> 原则：只保留当前里程碑必需依赖。未进入白名单的依赖默认不引入，发现冗余直接删除。

## 1. 当前基线（2026-02-07）

当前工程实际依赖保持最小集：

1. Web：`react`、`react-dom`、`vite`、`typescript`、`eslint` 及必要类型包；终端层已按白名单引入 `@xterm/xterm`、`@xterm/addon-fit`（`T-064`）；样式构建链使用 `tailwindcss` + `postcss` + `autoprefixer` + `@tailwindcss/postcss`（`T-074`）；中央角色画布与文件管理列表统一使用 `@tanstack/react-virtual`（`T-075`、`T-107`）；Git 面板引入 `@gitgraph/react` + `diff2html`（`T-109`）；文件编辑器使用 `codemirror` + 语言包（`T-114`）。
2. Tauri：`@tauri-apps/api`、`@tauri-apps/cli`。
3. Rust 侧能力依赖在 `Cargo.toml` 管理（不受本文件 npm 列表约束）。
   - 新增：`rusqlite`（`bundled`）用于本地 SQLite 存储。

结论：你提供的大清单目前绝大多数未被引入，不需要额外卸载动作。
补充决策：当前阶段不实现 Redis，相关 Node 侧 Redis 依赖默认禁止引入。

## 2. 允许引入白名单（按阶段）

### 2.1 P0（M1-M2 可引入）

1. 终端能力：`@xterm/xterm`、`@xterm/addon-fit`、`@xterm/addon-web-links`、`@xterm/addon-clipboard`
2. UI 组件：`@radix-ui/*`（按需）、`clsx`、`tailwind-merge`
3. 命令面板：`cmdk`
4. 拖拽布局：`react-resizable-panels`、`@dnd-kit/core`、`@dnd-kit/sortable`
5. 状态/请求：`@tanstack/react-query`、`zustand` 或 `jotai`（二选一）
6. 安全渲染：`dompurify`
7. Schema 校验：`zod`
8. 样式构建：`tailwindcss`、`postcss`、`autoprefixer`、`@tailwindcss/postcss`
9. 列表虚拟化：`@tanstack/react-virtual`

### 2.2 P1（V1 可引入）

1. 代码高亮：`shiki`
2. 代码编辑器：`codemirror`、`@codemirror/lang-javascript`、`@codemirror/lang-python`、`@codemirror/lang-rust`、`@codemirror/lang-json`、`@codemirror/lang-markdown`、`@codemirror/lang-css`、`@codemirror/lang-html`
3. Markdown 生态：`remark`、`remark-gfm`、`rehype-katex`（按需）
4. Diff 显示：`diff` 或 `@pierre/diffs` 或 `diff2html`（按场景二选一/三选一）
5. Git 图谱可视化：`@gitgraph/react`（仅 Git 协作视图启用）
6. 可视化图谱：`d3`（仅在 Agent Graph 落地时引入）
7. 错误监控：`@sentry/browser`（可选）

### 2.3 P2（V2+ 才考虑）

1. 协议适配：`@modelcontextprotocol/sdk`
2. 协作编辑：`yjs`、`@lexical/yjs`（只有实时协作确定后）
3. Redis/后端观测 Node 依赖（只在服务进程方案成立后引入）

## 3. 明确不引入（当前阶段）

以下类别默认删除或禁止新增：

1. Electron 专用：`electron-*`、`@sentry/electron`（项目基线是 Tauri）。
2. 服务端 Web 框架：`express`、`body-parser`、`cors`、`router` 等。
3. 大量 Babel 底层包：`@babel/*`（Vite+TS 方案不需要手工引入）。
4. 大量 OpenTelemetry Node 插桩包（未建立 Node 服务端前不需要）。
5. 多套编辑器内核并存（`lexical` 与 `prosemirror` 同时引入会失控）。
6. 重复版本并存依赖（如同包多个版本号），除非有明确兼容理由。

## 4. 引入/删除流程（强制）

1. 新增依赖前先标注场景、替代方案、体积与维护成本。
2. 必须同步更新本文件“白名单”条目。
3. 若依赖未被使用或场景取消，直接删除：
   - `npm uninstall <pkg> --workspace <workspace>`
4. 删除后必须验证：
   - `npm run typecheck`
   - `npm run build:web`
   - `cargo check --workspace`

## 5. 与你给的大清单的处理结论

1. 当前不整体引入该清单。
2. 仅按本文件白名单按需挑选。
3. 非白名单条目视为“默认不需要”，后续发现被误加将直接删除。
