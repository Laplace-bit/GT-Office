# GT Office 响应式样式单位重构方案

> 任务建议 ID：`T-187`
>  
> 文档目标：为 GT Office 前端样式系统提供一套可执行的“固定 `px` -> 响应式单位”重构方案，覆盖架构、通用单位转换、跨平台适配、实施步骤与验证标准，确保 Windows/macOS 桌面端在不同窗口尺寸和系统缩放下都保持稳定显示，并避免单位替换后视觉效果发生明显跳变。

## 1. 背景与目标

GT Office 当前前端样式系统已完成 SCSS 分层，但尺寸体系仍主要依赖固定 `px`，导致以下问题：

1. 不同窗口尺寸下布局弹性不足，面板和卡片在窄宽度时容易挤压或裁切。
2. Windows 高 DPI 缩放与 macOS Retina 下的密度表现不一致。
3. 左右 pane resize、设置页、Git、任务中心等复杂页面在收窄时容易出现横向滚动。
4. 设计令牌未形成“语义尺寸 -> 响应式单位”的统一映射，feature 仍在局部直接写死尺寸。

本次重构目标：

1. 将 `apps/desktop-web/src` 下业务 UI 的固定 `px` 单位全部迁移为响应式单位。
2. 建立统一的响应式设计令牌、断点体系、平台变量和尺寸语义层。
3. 保持默认窗口下的视觉比例与信息密度基本不变，避免“改单位后整体变大或变小”。
4. 支持 Windows/macOS 桌面端跨尺寸适配，重点覆盖 `768 / 1024 / 1280 / 1440 / 1728` 宽度场景。

范围约束：

1. 业务 SCSS、设计令牌、组件内联样式、UI 偏好中的字体尺寸映射全部纳入。
2. `node_modules` 和第三方库自身内部样式不纳入“零 `px`”目标。
3. 我方对第三方 DOM 的业务覆盖样式仍需清除 `px`。

## 3. 设计原则

结合仓库现有架构和 `$ui-ux-pro-max` 的建议，本次重构遵循以下原则：

1. 保持“专业桌面工作台”气质，优先稳定、克制、高信息密度，而不是做移动端网页化改造。
2. 先保证默认窗口和默认字体档位下视觉比例接近现状，再逐步引入流式尺寸。
3. Viewport 级布局变化只在 shell 层治理，feature 内部优先使用 container query。
4. 所有尺寸必须先进入语义 token，再由 token 映射到 `rem / clamp / % / fr` 等单位。
5. 平台差异必须收敛为平台 token，不允许 feature 自己继续判断 `windows / macos` 写死样式。
6. 所有重构必须以“无横向滚动、无裁切、可重排”为验收基线。

## 4. 响应式样式架构

### 4.1 现有样式层级

当前入口为：

- `apps/desktop-web/src/styles/index.scss`

当前层级为：

1. `tokens`
2. `foundations`
3. `features`
4. `utilities`

该层级可继续沿用，但需补充响应式尺寸体系。

### 4.2 建议新增文件

建议新增以下 token 文件：

1. `apps/desktop-web/src/styles/tokens/_responsive.scss`
2. `apps/desktop-web/src/styles/tokens/_platform.scss`
3. `apps/desktop-web/src/styles/tokens/_semantic-size.scss`

职责划分：

#### `tokens/_responsive.scss`

负责：

1. 统一断点。
2. `rem/clamp()` helper。
3. 流式尺寸公式。
4. viewport 与 container 的公共响应式 mixin。

#### `tokens/_platform.scss`

负责：

1. Windows/macOS 平台变量。
2. 原生标题栏安全区变量。
3. caption button / traffic-light 预留区尺寸。
4. 平台密度补偿系数。

#### `tokens/_semantic-size.scss`

负责：

1. spacing token。
2. radius token。
3. icon / control / pane / modal 尺寸 token。
4. hairline / blur / shadow 偏移等语义值。

### 4.3 架构规则

1. feature 层禁止继续定义裸尺寸常量。
2. 业务组件只能消费语义 token。
3. viewport 级断点只允许出现在 shell 层和全局 overlay 层。
4. feature 内部优先用 container query 做局部重排。
5. 不允许继续新增零散 `@media (max-width: 880px)` 之类的孤立断点。

## 5. 单位体系与禁用规则

### 5.1 允许单位

1. `rem`
2. `em`
3. `%`
4. `fr`
5. `minmax()`
6. `clamp()`
7. `dvh / svh / lvh`
8. `lh`

### 5.2 禁止单位

1. 业务源码中的 `px`
2. 业务 TSX 内联样式中的 `px`
3. feature 私自定义的新尺寸 magic number

### 5.3 单位使用约定

1. `rem`：字体、间距、圆角、边框、阴影偏移、blur、图标、控件尺寸
2. `em`：随文字缩放的局部图标和徽标
3. `% / fr / minmax()`：布局、分栏、列表、面板
4. `clamp()`：字号、gap、pane 宽度、modal 宽高、标题栏高度
5. `dvh`：全屏高度和主要壳层布局
6. `lh`：文本区域和说明块的垂直节奏

## 6. 通用单位转换方案

本节是本次方案的关键。目标不是机械替换单位，而是在迁移后尽量保持当前视觉比例，避免出现明显跳变。

### 6.1 转换基线

当前 GT Office 默认字体基线是 `14px`，而不是浏览器默认 `16px`。因此：

1. 本次 `px -> rem` 的默认换算基线采用 `14`。
2. 所有静态尺寸先按当前默认视觉等比还原，再视需要升级为流式尺寸。

基础换算函数：

```scss
@function px-to-rem($px, $base: 14) {
  @return calc(#{$px} / #{$base}) * 1rem;
}
```

示例：

| 旧值 | 新值 |
|---|---|
| `14px` | `1rem` |
| `13px` | `0.9286rem` |
| `12px` | `0.8571rem` |
| `24px` | `1.7143rem` |
| `280px` | `20rem` |

这样可以在默认 `medium` 字体档位下，尽量维持当前视觉尺寸感知不变。

### 6.2 分类转换策略

不同类型的尺寸必须使用不同策略，不允许统一机械替换。

#### A. 字体、间距、圆角、按钮高、图标尺寸

规则：

1. 先按 `14px` 基线转为 `rem`
2. 不主动改视觉比例
3. 纳入语义 token

适用示例：

1. `font-size`
2. `padding`
3. `gap`
4. `border-radius`
5. `width/height` 用于 icon、按钮、tag

#### B. 布局宽度、高度、卡片和弹窗尺寸

规则：

1. 不直接只用 `rem`
2. 优先改为 `clamp(min, ideal, max)`
3. 结合 `% / vw / fr`

适用示例：

1. 侧栏宽度
2. modal 宽度
3. overlay 宽高
4. 左右 pane 比例

#### C. 边框和 hairline

规则：

1. 不再出现 `1px`、`0.5px`
2. 统一进入 token

建议：

```scss
:root {
  --vb-border-width: 0.0625rem;
  --vb-hairline: 0.03125rem;
}
```

#### D. Blur、shadow、translate 偏移

规则：

1. 按原值等比换成 `rem`
2. 不调整视觉力度
3. 如后续需平台差异，再通过 token 覆盖

#### E. 视口高度

规则：

1. `100vh` 改为 `100dvh`
2. shell 和 overlay 优先使用 `dvh`
3. 标题栏安全区 fallback 使用 `0rem`，不用 `0px`

### 6.3 防视觉跳变规则

为避免更换单位后出现“整体变大、变小、变松、变挤”的问题，实施时必须遵守以下规则：

1. 第一步只做等比替换，不做视觉重设计。
2. 默认窗口、默认字体档位下，改造前后组件尺寸差异应控制在小范围内。
3. `clamp()` 的最小值不低于旧值的 `0.9x`，最大值不高于旧值的 `1.15x`。
4. 同一组件内的 icon、文字、padding、radius 必须按同一比例组转换。
5. 不允许只把外层容器改成响应式，内部子元素仍按旧比例不动。

推荐流式函数：

```scss
@function fluid-rem($min-px, $ideal-vw, $max-px, $base: 14) {
  $min-rem: calc(#{$min-px} / #{$base}) * 1rem;
  $max-rem: calc(#{$max-px} / #{$base}) * 1rem;
  @return clamp(#{$min-rem}, #{$ideal-vw}, #{$max-rem});
}
```

### 6.4 语义尺寸映射

禁止“看到多少 `px` 就改多少 `rem`”。必须先进入语义层。

建议最小语义集合：

```scss
:root {
  --vb-space-2xs: 0.2857rem;
  --vb-space-xs: 0.4286rem;
  --vb-space-sm: 0.5714rem;
  --vb-space-md: 0.8571rem;
  --vb-space-lg: 1.1429rem;
  --vb-space-xl: 1.7143rem;

  --vb-radius-xs: 0.4286rem;
  --vb-radius-sm: 0.5714rem;
  --vb-radius-md: 0.8571rem;
  --vb-radius-lg: 1.1429rem;
  --vb-radius-xl: 1.4286rem;

  --vb-size-icon-sm: 0.8571rem;
  --vb-size-icon-md: 1rem;
  --vb-size-icon-lg: 1.1429rem;

  --vb-size-control-sm: 1.7143rem;
  --vb-size-control-md: 2rem;
  --vb-size-control-lg: 2.2857rem;

  --vb-pane-width-sm: clamp(14rem, 22vw, 18rem);
  --vb-pane-width-md: clamp(16rem, 25vw, 20rem);
  --vb-modal-width-md: clamp(34rem, 62vw, 48rem);
}
```

上述数值不是新视觉方案，而是对当前系统的等比抽象，目的在于降低重构期间的视觉波动。

### 6.5 UI 字体档位兼容

当前 `ui-preferences.ts` 中的字体档位使用固定 `px`：

1. `small: 13px`
2. `medium: 14px`
3. `large: 15px`
4. `xlarge: 16px`

改造后仍保留这四个枚举，但不再返回 `px` 字面量：

| 档位 | 新值 |
|---|---|
| `small` | `0.9286rem` |
| `medium` | `1rem` |
| `large` | `1.0714rem` |
| `xlarge` | `1.1429rem` |

这样既能去除 `px`，也能维持用户对字体偏好的感知一致性。

## 8. Windows / macOS 响应式适配架构

### 8.1 平台变量收口

当前平台标识已通过 `data-vb-platform` 写入根节点，可保留该机制，但尺寸差异必须统一下沉为 token。

建议新增：

```scss
:root {
  --vb-platform-topbar-height: clamp(2rem, 2vw, 2.5rem);
  --vb-platform-leading-inset: 0rem;
  --vb-platform-window-control-width: 0rem;
  --vb-platform-pane-min-width: 14rem;
  --vb-platform-density-scale: 1;
}

:root[data-vb-platform='windows'] {
  --vb-platform-window-control-width: clamp(6rem, 7vw, 7.5rem);
  --vb-platform-pane-min-width: 14.1429rem;
}

:root[data-vb-platform='macos'] {
  --vb-platform-leading-inset: clamp(4rem, 5vw, 5rem);
  --vb-platform-density-scale: 0.98;
}
```

### 8.2 Windows 重点

需重点处理：

1. `100% / 125% / 150%` DPI 缩放
2. caption button 区域宽度
3. pane 最小宽度
4. 字号密度
5. scrollbar/thumb 视觉密度

目标：

1. 高缩放下 top bar 不挤压
2. 面板文字不换行失控
3. pane resize 后仍保持稳定重排

### 8.3 macOS 重点

需重点处理：

1. 原生标题栏安全区
2. traffic-light 预留区
3. 玻璃 blur 和圆角的密度一致性
4. Retina 下的视觉节奏

规则：

1. `env(titlebar-area-height, 0px)` 改为 `env(titlebar-area-height, 0rem)`
2. 不允许 feature 再单独判断 `macos` 写局部尺寸
3. top bar 的 padding、leading inset 和 hover 行为全部收口到平台 token

#### 4.1 Git feature 的分块迁移顺序

Git 模块是本轮最先落地的样板，因为它同时覆盖列表虚拟化、表单控件、历史视图和 diff/graph 可视化，能验证“固定像素 -> 语义尺寸 -> 流式布局”的完整链路。

建议按以下顺序迁移：

1. `operations`
2. `history`
3. `diff`
4. `graph`

分块原则：

1. 先迁移 shell 级头部、操作区、过滤区和表单区，再处理高密度列表和可视化区。
2. 先保证交互密集区可读可点，再优化视觉细节。
3. 任何块的完成标准都必须包含“窄宽度下无横向滚动”和“组件内重排可用”，不能只看样式文件有没有去掉 `px`。

#### 4.2 `operations` 迁移策略

范围：

1. Git 面板头部
2. 分支状态摘要
3. 操作按钮组
4. `changes` 分区
5. `commit` / `branches` / `stash` 三个折叠区

迁移要点：

1. 头部从“固定左右布局”改为“可换行的双层布局”，窄宽度下让操作按钮自然下沉到下一行。
2. 分区标题、count、filter chip、action button 统一使用语义 token，不再由局部 `px` 决定密度。
3. `changes` 区的高度不再依赖像素常量硬编码，而是按可见内容容器的实际高度，映射成 rem 或 CSS 变量后再喂给布局。
4. 表单输入框、select、按钮组的最小高度必须对齐同一 control token，避免“文本区变大但按钮还是旧高度”的不协调问题。

#### 4.3 `history` 迁移策略

范围：

1. 历史切换开关
2. `GitGraphView`
3. commit detail 面板
4. 详情区文件列表

迁移要点：

1. 历史视图必须从“固定表格宽度”改成“可压缩的列模板”，图表列、描述列、日期列、作者列、hash 列分别进入 token 或 CSS 变量。
2. commit detail 面板要在窄宽度下自动堆叠，避免标题、hash、作者、日期挤在一行里。
3. 文件列表采用列数收缩策略，窄宽度下允许从三列收敛为单列或双列。
4. 只要历史页进入 container query 命中范围，就应优先压缩非核心列，而不是强行维持桌面宽表格。

#### 4.4 `diff` 迁移策略

范围：

1. unified diff
2. split diff
3. diff header / action footer
4. diff body line-height / gutter / code block

迁移要点：

1. diff 区的行高、gutter 宽度、按钮尺寸统一进入 token。
2. code body 不再依赖像素字号描述行距，应使用 `rem` + `line-height` 的语义组合。
3. split diff 在窄宽度下必须默认让内容先可读，不强求双栏同时展开。
4. 当可用宽度不足时，优先把辅助信息折叠到 header 或 tooltip，不要保留固定宽度的多列并引发横向滚动。

#### 4.5 `graph` 迁移策略

范围：

1. graph header
2. lane 渲染区
3. commit row
4. refs badge
5. selected commit detail

迁移要点：

1. graph 列宽、行高、日期列、作者列、hash 列都必须有统一的 token 语义，避免 TSX 里重复写像素宽度。
2. 虚拟列表的 `row height`、`min width`、`translateY` 必须从像素字符串迁移到 rem 计算结果，或者由 CSS 变量承载后再由虚拟化逻辑读取。
3. lane 和 node 的图形尺度可以保留视觉比例，但必须由同一个尺寸基线驱动，不能一边 rem 一边 px。
4. 详情区在窄宽度下优先纵向堆叠，refs badge 允许提前截断，避免压缩主标题和 hash。

#### 4.6 Git 迁移的完成判定

## 12. 风险与默认决策

### 12.1 风险

1. 直接机械替换单位会造成视觉比例失衡。
2. container query 与现有 viewport media query 混用可能引发规则冲突。
3. 第三方 DOM 覆盖样式可能需要额外兼容处理。

### 12.2 默认决策

1. 所有 `px -> rem` 转换以当前 `14px` 根字号为基线。
2. 先等比还原，再引入 `clamp()` 流式尺寸。
3. feature 不单独定义平台差异和断点。
4. 本次不改后端 command/event 契约。
5. Linux 不作为主验收矩阵，但不得明显回归。
6. Git feature 先作为样板模块完成，再用同一套 token 和 helper 推广到 Settings / Task Center / Workspace Hub。
