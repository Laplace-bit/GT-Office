# AGENTS.md

本文件定义 `GT Office` 项目的开发规范与协作约束，适用于人类开发者与 AI Coding Agent。

## 1. 项目目标

- 打造跨平台（Windows/Linux/macOS）高性能 AI Coding 桌面工具。
- 核心能力：Workspace、文件管理、真终端、Git、多窗口并行、Tool Adapter、Change Feed。

## 2. 技术基线

- 框架：Tauri + Rust + WebUI（React/Svelte）。
- Rust：`tokio`、`portable-pty`、`notify`、`git2`、`tracing`。
- 前端：`xterm.js`、Tailwind、状态管理轻量化（zustand/jotai）。
- 数据：SQLite + `.gtoffice/config.json`。
- 配置：默认配置 + 用户配置 + 工作区配置 + 会话覆盖的分层模型。

## 3. 目录职责约束

- `docs/`：需求、架构、进度、交接文档。
- `apps/`：UI 与 Tauri 壳层。
- `crates/`：Rust 领域模块。
- `packages/`：共享类型与可复用组件。
- `tests/`：集成与端到端测试。

禁止事项：

1. 将业务逻辑直接写进 Tauri 命令入口。
2. 让 UI 直接依赖系统能力实现。
3. 跨模块随意引用，破坏边界。

## 4. 开发流程

1. 开发前先确认 `docs/01` 与 `docs/02` 是否已覆盖需求。
2. 功能编码前必须补齐 `docs/05`（用户故事、主/异常流程、验收用例）。
3. 前后端联调前必须确认 `docs/06`（命令、事件、错误码）无歧义。
4. 编码完成后更新 `docs/03` 的任务状态。
5. 会话结束前更新 `docs/04` 最近交接快照。
6. 所有改动需可追溯到任务 ID（如 `T-00x`）。

## 5. Agent 交接与无缝续接（强制）

### 5.1 会话结束时必须完成

1. 更新 `docs/04_上下文交接文档.md` 的“最近交接快照”。
2. 更新 `docs/03_项目开发进度跟踪.md` 的任务状态与变更记录。
3. 输出“Next Agent Starter”交接块，至少包含：
   - 当前里程碑与本轮目标
   - 已完成与未完成（必须具体到第一步动作）
   - 改动文件清单
   - 风险/阻塞与是否需要决策
   - 已执行验证命令与结果
4. 若未执行验证，必须写明“未验证”及原因，禁止省略。

### 5.2 下一个 Agent 启动时必须执行

1. 按顺序读取：`docs/README.md` -> `docs/03_项目开发进度跟踪.md` -> `docs/04_上下文交接文档.md` -> `docs/05_高质量功能设计_核心工作流.md` -> `docs/06_API与事件契约草案.md` -> `AGENTS.md`。
2. 复述当前目标、第一执行动作、成功标准后再开始改动。
3. 若“最近交接快照”缺失关键信息，先补齐交接快照再继续开发。
4. 严禁使用“见上文/参考之前内容”作为交接结论，必须落到结构化条目。
5. 完成本轮任务后，必须重复执行 `5.1`，更新 `docs/03` 与 `docs/04` 后再结束会话。

### 5.3 交接质量红线

1. 不允许只写总结，不写可执行下一步。
2. 不允许只写文件名，不写改动目的与影响范围。
3. 不允许只写“测试通过”，必须写命令或验证路径。
4. 多工作区场景必须注明目标 `workspace_id` 或目标目录。
5. `docs/04` 只保留“上一任 -> 下一任”的直接交接，不累计长历史。

## 6. 需求设计质量门槛（必须满足）

每个功能点必须输出以下设计要素，不满足则不能进入实现：

1. `FR`（功能需求）与 `NFR`（非功能指标）明确且可验收。
2. 至少 1 条 User Story（包含前置条件、主流程、异常流程）。
3. 接口契约：命令入参/出参、事件 payload、错误码。
4. 数据影响：涉及表结构、配置字段或状态机变化需标注。
5. 验收用例：至少 1 条正常流 + 1 条异常流。

## 7. 代码规范

### Rust

1. 使用 `clippy` 与 `rustfmt`。
2. 错误处理统一使用领域错误类型，禁止 `unwrap()` 滥用。
3. 关键链路必须有 `tracing` 日志。

### Frontend

1. 组件按 feature 分层，避免全局状态泛滥。
2. 大列表必须虚拟化。
3. 主题、快捷键、可访问性为默认能力。

## 8. 工程架构规范

1. 核心能力必须通过抽象接口暴露，禁止上层直接依赖基础设施实现。
2. 多工作区相关操作必须显式携带 `workspace_id`，不得依赖隐式全局状态。
3. 终端会话创建默认 `cwd = workspace.root`，自定义 cwd 需通过越界校验。
4. Rust 优先使用 trait + 组合实现多态扩展，避免不可控的深层继承链。
5. Provider 实现必须支持 mock，以便对 workspace/terminal/git/settings 做隔离测试。

## 9. 安全规范

1. 默认命令仅允许在 workspace 内执行。
2. API Key 等密钥必须走系统凭据存储，不允许明文落盘。
3. 高危命令和高风险文件操作必须二次确认。
4. AI 配置能力遵循白名单字段与最小权限原则。
5. AI 改配置必须走“预览 -> 校验 -> 确认 -> 应用 -> 审计”流程。

## 10. UI 设计规范

1. 视觉风格遵循苹果风格：简洁、高级、专业、强秩序。
2. 使用统一设计令牌（色彩、间距、圆角、阴影、动效、字体层级）。
3. 动效以低干扰、快速反馈为原则，不得影响高频操作效率。
4. 设置页、终端页、工作区页必须保持一致的组件语言与交互语义。
5. 默认字体建议：UI 使用 `SF Pro` 风格，代码区使用高可读等宽字体。

## 11. Git 与提交流程

1. 分支命名：`feat/*`、`fix/*`、`refactor/*`、`docs/*`。
2. Commit 规范：Conventional Commits（如 `feat:`, `fix:`, `docs:`）。
3. PR 必须包含：变更说明、风险、验证步骤。

## 11.1 依赖治理（强制）

1. 依赖选型以 `docs/07_依赖选型与精简清单.md` 为准，未入白名单默认不引入。
2. 新增依赖必须说明用途、替代方案、影响范围，并同步更新 `docs/07`。
3. 发现未使用或非白名单依赖时，直接删除，不做“先保留”。
4. 依赖变更后必须执行最小验证：`npm run typecheck`、`npm run build:web`、`cargo check --workspace`。

## 12. Definition of Done（DoD）

一个任务标记 `DONE` 前必须满足：

1. 功能满足对应需求与验收标准。
2. `docs/05` 与 `docs/06` 已同步更新（若有契约或流程变化）。
3. 关键路径有测试或可复现实验步骤。
4. 文档已更新（至少 `docs/03`，必要时 `docs/01/02/04`）。
5. 无阻塞性已知缺陷未记录。

## 13. AI 协作规则

1. AI 修改代码前，先读取相关文档并声明目标文件。
2. AI 需优先给出可执行改动，不只给建议。
3. AI 做需求设计时，必须输出结构化内容，不可只给概念性总结。
4. AI 结束会话时必须输出：改动文件列表、验证结果、下一步建议。
5. AI 完成任务后必须更新 `docs/03` 与 `docs/04`，否则不得结束会话。

## 14. 文档优先级

若文档冲突，优先级如下：

1. `docs/01_需求与产品设计.md`
2. `docs/05_高质量功能设计_核心工作流.md`
3. `docs/06_API与事件契约草案.md`
4. `docs/02_系统架构与模块目录设计.md`
5. `AGENTS.md`
6. 其他说明文档
